# Agent 竞品调研

## 引言

过去一年，Agent 明显从“能聊几句的模型能力”走向了“能接任务、能调用工具、能持续推进工作”的产品形态。

这也让竞争的重点发生了变化。大家表面上比的还是模型、工具链、代码能力，但真正拉开差距的，已经不只是“单次结果有多强”，而是：

- 用户愿不愿意持续把任务交给它
- 用户是否相信它能在真实场景里稳定推进工作
- 用户和它协作一段时间后，是否会形成可积累的默契

所以这篇调研想讨论的，不只是 Agent 做了哪些功能，而是一个更核心的问题：

> 好的 Agent 产品，到底是怎样一步步建立用户信任，最后变成“值得长期协作的对象”。

围绕这个问题，本文重点分析三类头部产品：`OpenAI Codex`、`Claude Code`、`Gemini CLI`。我会从产品战略、体验设计和商业模式三个层面展开，并尽量把结论写清楚，方便后续直接作为设计讨论的输入。

## 1、Agent 产品概览

> **本章结论**
> Agent 行业已经从“能力展示”进入“结果交付”阶段。往后真正有竞争力的，不只是模型强，而是能不能把模型、工具、工作流和产品体验组织成一个稳定的任务承接系统。

随着工具调用、多步推理、长上下文和环境感知能力逐渐成熟，AI 产品正在从“提供能力”转向“交付结果”。

从能力层面看，GPT、Gemini、Claude 这一代模型已经不只是生成内容，还能做任务规划、工具调用和自我修正，这让 Agent 具备了落地的基础。

从商业层面看，Agent 也正在改变 AI 产品的价值结构。过去用户为“模型能力”买单，未来更可能为“任务完成度”和“协作效率”买单。谁能承接更完整的任务链路，谁就更容易建立留存、付费和生态绑定。

按能力栈和产品形态看，当前行业里的 Agent 大致可以分成五类：

| 分类 | 描述 | 代表产品 |
| --- | --- | --- |
| 模型 + Agent 一体化 | 模型能力和产品形态深度耦合，目标不是做一个外挂工具，而是把 Agent 做成核心产品形态 | Gemini、OpenAI Codex、Claude Code |
| Agent 平台 | 提供任务编排、工具调用、记忆、多 Agent 协同等基础设施，方便开发者搭建自己的 Agent | LangChain、LangGraph、Dify、Coze |
| AI 原生 IDE / 编辑器 | 在编辑器或 IDE 内重构代码协作体验，把 Agent 编织进开发流程 | Cursor、Windsurf、Zed AI |
| 通用执行型 Agent | 强调“你提任务，我来完成”，弱化配置和调度过程 | Manus、Devin、Lovable |
| 垂类 Agent 工具 | 聚焦单一场景深度优化，用稳定、直接的方式解决具体问题 | Perplexity、Notion AI、Otter.ai |

这里需要说明两点：

- Cursor、Windsurf 更适合归到“AI 原生 IDE / 编辑器”，而不是广义的 Agent 平台，因为它们绑定的是代码编辑场景，不是给开发者做中台能力。
- 原稿中提到的 `OpenClaw` 暂时没有明确对应到公开产品，建议后续再核对一次名称。

**小结**

这一轮 Agent 竞争，已经不是单点能力的竞争，而是“谁能把能力组织成一个可持续协作系统”的竞争。后文对竞品的分析，也会尽量围绕这个角度展开。

## 2、核心竞品选择

> **本章结论**
> 如果把 DeepSeek 的潜在路径理解为“自研模型 + Agent 应用一体化”，那么最值得重点研究的，不是泛 Agent 平台，也不是单纯的 IDE 工具，而是已经把模型能力和产品体验捏在一起的代表性产品。

基于这个判断，后续重点选择三家作为核心竞品：

- `OpenAI Codex`
- `Claude Code`
- `Gemini CLI`

选择它们的原因很直接：

- 三者都不是单纯的“聊天入口”，而是明确在往任务执行和协作系统演进
- 三者都把模型能力、工具链和使用体验做了较深的一体化
- 三者在战略选择上差异明显，适合做横向对比

**小结**

这三家并不代表全部 Agent 路线，但它们很能代表目前头部玩家在“如何把 Agent 做成长期产品”这件事上的不同做法。

## 3、竞品分析

> **本章结论**
> 三家产品虽然路径不同，但都在回答同一个问题：怎样让用户敢把更重要、更完整的任务长期交给 Agent。差异主要体现在战略定位、协作体验和商业化方式上。

这一章从三个角度展开：

- `产品战略`：它想成为什么样的产品
- `产品体验`：它是怎么建立用户信任的
- `商业模式`：它如何把价值转成可持续收入

---

### 3.1 产品战略对比

> **本节结论**
> 三家的战略方向很不一样，但共性非常清楚：都在把 Agent 从“工具”推向“协作者”。区别不在于都想做大，而在于各自打算从哪里切入、服务谁、把壁垒建在哪。

#### 1. 三种不同的发展方向

| 维度 | Codex | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 产品定位 | 以云端为核心，面向更完整的 AGI 产品形态延展 | 强调治理、能力和组织协作的一体化，像“可管理的数字员工” | 以开源和终端优先为基础，做低门槛、可扩展的开发者协作终端 |
| 目标用户 | 从个人开发者到企业团队的广泛用户 | 高价值企业客户、重度开发者、治理要求高的组织 | 个人开发者、学生，以及重视开放性和成本的用户 |
| 初始形态 | 云端平台，与 ChatGPT 深度联动 | 终端工具，与本地工作环境深度融合 | 开源 CLI，与 Gemini Code Assist 打通 |
| 演进方向 | 扩大生态覆盖面，打通终端、IDE、Web、移动、GitHub 等触点 | 提升自主性、可治理性和组织嵌入能力，成为工程体系中的正式角色 | 借助扩展、subagents、hooks、browser agent 等能力，走开放生态路线 |

如果用一句话概括：

- `Codex` 走的是“平台化 AGI 入口”路线
- `Claude Code` 走的是“企业级数字员工”路线
- `Gemini CLI` 走的是“开源开发者协作终端”路线

这三条路背后的差异，其实是对“护城河”理解不同：

- Codex 更相信生态整合能力
- Claude Code 更相信流程嵌入和治理深度
- Gemini CLI 更相信开放、低成本和用户规模

#### 2. 三家的共性趋势

把它们的更新动作放在一起看，会发现三家虽然风格不同，但补的能力很像。我把这些共性归纳成四条主线：

1. `更联通`：Agent 不再是单一入口，而是嵌进更多工作触点
2. `更懂我`：Agent 不只是理解 prompt，而是越来越理解环境和上下文
3. `更靠谱`：Agent 要能进入真实生产流程，而不只是做 demo
4. `任务上限更高`：Agent 要能承接更长、更复杂、更接近真实工作的任务

这四条主线，本质上对应着四类用户信任：

- 更联通，对应的是“我能不能方便地用到你”
- 更懂我，对应的是“你是不是越用越理解我”
- 更靠谱，对应的是“你交付的结果我敢不敢直接接”
- 任务上限更高，对应的是“我能不能把更重要的事交给你”

#### 3. 四条趋势分别意味着什么

##### 趋势一：更联通

**关键变化**

Agent 正在从独立的对话入口，变成覆盖 IDE、代码托管平台、项目协作工具和自动化系统的统一代理。

**对用户的意义**

这会明显降低迁移成本。用户不用频繁切工具、搬上下文，也更容易把 Agent 纳入现有工作流。

**一句话总结**

联通性决定了 Agent 能不能成为“顺手的工具”，也是长期协作关系的起点。

*见附录表格 1*

##### 趋势二：更懂我

**关键变化**

Agent 的“理解”正在从 prompt 本身，扩展到真实项目状态、环境上下文和可持续记忆系统。

**对用户的意义**

解释成本会下降，用户能更快拿到贴合现场的问题判断和解决方案；同时，跨任务和跨阶段的协作连贯性也会更强。

**一句话总结**

真正拉开长期体验差距的，往往不是第一次回答，而是第十次合作时它还记不记得你之前怎么做事。

*见附录表格 2*

##### 趋势三：更靠谱

**关键变化**

Agent 不再停留在“帮忙写点代码”的层面，而是往权限、审计、审批、可观察性和安全治理这些更接近生产环境的能力上补。

**对用户的意义**

用户不是只想要一个 demo，而是希望拿到能进入真实流程的交付物。越靠近生产标准，用户越敢放手使用。

**一句话总结**

可靠性不是锦上添花，而是 Agent 从“能用”走向“敢用”的分水岭。

*见附录表格 3*

##### 趋势四：任务上限更高

**关键变化**

复杂任务越来越依赖“计划 - 拆解 - 执行 - 验证 - 修正”的完整循环，而不是单轮对话直接给答案。

**对用户的意义**

Agent 能承接的事情越完整，用户就越容易把它视为协作对象，而不是只在局部时刻调用的工具。

**一句话总结**

任务上限决定了 Agent 在真实工作中能占多大比重，也决定了它能不能成为值得长期托付的角色。

**本节小结**

战略上看，三家路径不同，但最终都在朝同一个方向走：让用户不仅“会用”，还“敢交付”。这也是 Agent 产品和传统 AI 工具最根本的区别。

### 3.2 产品体验对比

> **本节结论**
> Agent 体验的核心，不只是功能是否完整，而是它有没有把“协作关系”设计出来。用户不是只在判断它聪不聪明，也在判断它是否值得合作、是否可控、是否能长期信任。

我更倾向于把 Agent 体验理解成“关系设计”。

这里面至少有两层能力：

- `硬实力`：它能不能把任务做好
- `软实力`：它怎么沟通、怎么反馈、怎么让人放心

顺着这个思路，可以把用户和 Agent 建立关系的过程拆成五个阶段。

#### 1. 体验分析框架

| 用户阶段 | 用户心里在判断什么 | 一个好的 Agent 应该给出的感受 | 常见流失点 |
| --- | --- | --- | --- |
| 第一印象 | “我能不能快速理解你有什么用” | 定位清楚、门槛不高、容易上手 | 看不懂、不会用、觉得离自己太远 |
| 任务测试 | “我敢不敢把事交给你做” | 过程透明、边界清楚、随时可介入 | 黑盒执行、没有反馈、缺少留痕 |
| 结果验收 | “你交出来的东西行不行” | 结果可读、可用、可继续改 | 结果差、看不懂、无法判断质量 |
| 问题迭代 | “你能不能按我的意思修” | 理解准确、修改稳定、版本有记录 | 反复解释、越改越偏、无法回退 |
| 长期协作 | “继续和你合作值不值” | 越用越懂我、成本可接受、收益能积累 | 成本高、迁移难、长期收益不明显 |

这个框架既适合看桌面端，也适合看 CLI。不同的是，桌面端更考验“上手友好”和“情绪摩擦”，CLI 更考验“效率、可控性和可编排性”。

---

#### 2. 桌面端体验：Codex vs Claude Code

> **桌面端一句话结论**
> Codex 更像一个顺手、亲和、容易开始合作的助理；Claude Code 更像一个专业、可治理、值得长期磨合的工程同事。

##### 阶段一：初次上手

**共性**

- 两者界面都比较克制，主功能区和辅助区划分清楚
- 都尽量降低首次理解成本，让用户能迅速进入任务状态

**差异**

| Codex | Claude Code |
| --- | --- |
| 更像面向大众用户的产品，GUI 封装更完整，入口更友好 | 更像开发工作台，保留了更多 IDE 风格和工程语境 |
| 很多底层能力被产品层包装好了，用户不需要先理解复杂概念 | 信息密度更高，但“专业感”也来得更直接 |
| 工作栏命名更日常，例如 `project`、`chat` | 工作栏术语更开发者化，例如 `session` |
| 适合“不想先学一堆规则就开始用”的人 | 适合“愿意理解工具结构，再逐步用深”的人 |

**这一阶段的判断**

Codex 更会降低第一步的心理门槛；Claude Code 更擅长建立“这是个专业工具”的认知。

##### 阶段二：任务执行

**共性**

- 执行前会确认不明确的信息
- 执行中都有过程透出、任务拆分和留痕
- 遇到权限问题时会主动询问
- 支持中途打断和介入

**差异**

| Codex | Claude Code |
| --- | --- |
| 默认更省心，确认更少，节奏更流畅 | 默认更谨慎，确认更多，边界感更强 |
| 环境感知更主动，用户补充操作更少 | 用户需要更多显式输入和介入 |
| 留痕有，但颗粒度相对粗一些 | 留痕更细，任务拆解和分支信息更适合追踪 |
| 对话反馈更柔和，情绪价值更足 | 对话风格偏克制，但可靠感更强 |

**这一阶段的判断**

Codex 更像“会替你省事”的人；Claude Code 更像“每一步都留记录”的人。

##### 阶段三：结果验收

**共性**

- 都能交付可用结果
- 完成后都支持预览和进一步校准

**差异**

| Codex | Claude Code |
| --- | --- |
| 更强调“持续在场感”，像一直待命的助手 | 更强调“工作记录和监控感”，像有交班机制的同事 |
| 对话入口常驻，方便用户随时继续接着聊 | 工作台感更强，用户主导权更明显 |

**这一阶段的判断**

Codex 让人更容易产生“可以继续把事交给它”的感觉；Claude Code 则更容易让人觉得“它做过什么我都查得到”。

##### 阶段四：问题迭代

| Codex | Claude Code |
| --- | --- |
| 能配合修改，但长会话里更容易忘掉前文要求 | 上下文理解和记忆更强，越往后合作越稳定 |
| 更适合轻量、多轮但不特别长的迭代 | 更适合长期、复杂、需要反复打磨的任务 |

**这一阶段的判断**

如果任务链路很长，Claude Code 的记忆和一致性优势会更明显。

##### 阶段五：长期合作

| Codex | Claude Code |
| --- | --- |
| 成本感知相对更轻，整体更顺手 | token 压力更明显，但可调教深度也更高 |
| 生态一体化更强，适合 all-in-one 使用 | 工作台和规则体系更完整，适合深度定制 |
| 软性体验更好，沟通负担小 | 越用越懂你，但前期磨合成本更高 |

**桌面端小结**

如果目标是尽快建立协作、降低门槛，Codex 的体验更占优势；如果目标是长期沉淀工作方式、提升可治理性，Claude Code 的上限更高。

---

#### 3. CLI 端体验：Codex vs Claude Code vs Gemini CLI

> **CLI 端一句话结论**
> Codex 适合“马上开干”；Claude Code 适合“深度嵌入工作流”；Gemini CLI 适合“低成本长期试错，并且保持高度透明”。

CLI 场景里，用户对“亲和力”的容忍度没那么高，对效率、权限、可中断、自动化和可配置的敏感度会更高。所以同样是五阶段，重点会不太一样。

##### 阶段一：初次上手

**共性**

- 三家都支持“一条命令进入工作状态”
- 首启都提供最小可用的 onboarding
- 都支持通过项目级规则文件持续描述项目约束

**差异**

| 维度 | Codex CLI | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 首启感受 | 最顺手，产品化程度高 | 最像专业工具，信息密度高 | 最容易试，开源和免费策略降低心理门槛 |
| 安装门槛 | 低 | 中 | 很低 |
| 默认人设 | 已经很懂事的高阶 copilot | 严肃、专业的开发代理 | 透明、开放、适合试错的搭档 |

**这一阶段的判断**

Codex 负责“顺”，Claude Code 负责“专业”，Gemini CLI 负责“容易开始且没负担”。

##### 阶段二：任务执行

**共性**

- 都有权限模式和敏感操作确认
- 都支持流式过程输出和 diff 展示
- 都能中断，并保留必要上下文

**差异**

| 维度 | Codex CLI | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 权限模型 | 简洁直觉，三档就够用 | 粒度最细，治理能力最强 | 模式更多，兼顾灵活和隔离 |
| 过程透出 | 简明 | 最细 | 中等，但 plan 和 checkpoint 很实用 |
| 用户掌控感 | “我说做什么你就做什么” | “你的每个动作我都能管” | “我可以边试边改主意” |

**这一阶段的判断**

Codex 更强调效率，Claude Code 更强调控制，Gemini CLI 更强调可回退和低风险试验。

##### 阶段三：结果验收

**共性**

- 三家都支持 diff 交付
- 都支持 headless 或非交互执行
- 都能输出结构化结果，方便接入脚本或 CI

**差异**

| 维度 | Codex CLI | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 自动化友好度 | 强 | 最强 | 强 |
| 留痕颗粒度 | 中等 | 最细 | 中等，但快照能力强 |
| 结果后的在场感 | 等待下一条指令 | 留下完整工作记录 | 留下可回滚的任务快照 |

**这一阶段的判断**

Claude Code 最像能接进工程体系的同事；Gemini CLI 则把“反悔成本低”做得很突出。

##### 阶段四：问题迭代

| 维度 | Codex CLI | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 调教方式 | 规则文件为主 | 规则文件 + skills + subagents + hooks | 规则文件 + 分层记忆 + 可视化 memory |
| 调教成本 | 低 | 高 | 中 |
| 调教上限 | 中 | 最高 | 中高 |
| 调试可见性 | 一般 | 较好 | 最好 |

**这一阶段的判断**

Codex 的好处是简单；Claude Code 的好处是深；Gemini CLI 的好处是你能比较清楚地看到它到底依据什么规则在工作。

##### 阶段五：长期合作

| 维度 | Codex CLI | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 成本结构 | 与 ChatGPT 生态打通，已有订阅用户更划算 | 订阅和 token 成本更明显 | 免费额度友好，试错成本最低 |
| 生态绑定 | 中 | 中高 | 低 |
| 长期沉淀 | 靠规则文件积累 | 靠规则、subagents、hooks 等形成工作资产 | 靠记忆和透明规则积累 |
| 更适合谁 | 普通开发者、想直接干活的人 | 重度 CLI 用户、平台工程团队、企业场景 | 学生、独立开发者、偏爱开源的人 |

**CLI 端小结**

如果只看 CLI：

- 想少折腾、直接干活，优先 `Codex CLI`
- 想深度接入工作流、形成团队资产，优先 `Claude Code`
- 想低成本长期使用、又重视透明度和开放性，优先 `Gemini CLI`

**本节小结**

体验层面最值得注意的一点是：用户对 Agent 的判断从来不只是“这次结果好不好”，而是“我和它协作的时候，压力大不大、掌控感够不够、长期投入值不值”。这也是为什么同样是强模型，不同产品最后带来的关系感受会差很多。

---

### 3.3 商业模式对比

> **本节结论**
> 三家代表了三种很典型的商业逻辑：Codex 是“生态打包”，Claude Code 是“价值定价”，Gemini CLI 是“开源获客”。它们背后并不是简单的价格差异，而是对用户获取、产品定位和长期增长方式的不同判断。

先说结论：

- `Codex`：把 Agent 能力打包进 ChatGPT 订阅体系
- `Claude Code`：把 Agent 作为独立高价值产品收费
- `Gemini CLI`：用开源和免费额度扩大用户基数，再向上游生态转化

#### 横向对比

| 维度 | Codex | Claude Code | Gemini CLI |
| --- | --- | --- | --- |
| 核心定价思路 | 并入 ChatGPT 订阅体系 | 独立订阅 + 更清晰的额度分层 | 免费 tier + 订阅 / API 升级 |
| 用户转化路径 | 先成为 ChatGPT 用户，再自然使用 Codex | 从轻度到重度、从个人到团队逐层升级 | 先低成本尝试，再视需求升级 |
| 成本感知 | 平时不强，超额时才明显 | 最透明，用量感知强 | 免费期最轻，升级后再显现 |
| 长期 ROI | 对已有 ChatGPT 用户友好 | 对重度用户友好 | 对试错和规模获取友好 |
| 商业策略本质 | 生态绑定 | 价值定价 | 开源获客 |

#### 三家的模式分别说明了什么

##### Codex：生态打包

优点是门槛低，用户教育成本也低。对已经在 ChatGPT 体系里的用户来说，Codex 很容易变成“顺手就用”的能力。

问题在于，Agent 的单独价值感会被削弱。用户会知道它好用，但不一定会明确意识到“我是在为 Agent 本身付费”。

##### Claude Code：价值定价

优点是逻辑最清晰。它把“能托管多少任务、能承担多重的工作”转成了可感知的付费层级，适合把 Agent 当成正式生产力工具来卖。

问题是轻度用户会觉得贵，但这反过来也能帮助产品筛出真正愿意长期投入的高价值用户。

##### Gemini CLI：开源获客

优点是扩张效率高。免费额度和开源策略能快速吸引尝试者，特别适合建立开发者口碑和用户规模。

问题是直接变现压力会更大，需要依赖更上层的生态承接这些用户。

#### 对 DeepSeek 的启示

如果 Agent 是核心产品，应该更接近 Claude Code 的思路，让价值和价格关系更直接。

如果 Agent 是已有产品矩阵的延伸，可以参考 Codex 的打包方式，先做“低决策成本”的使用入口。

如果当前阶段更需要抢用户基数、抢开发者心智，可以借鉴 Gemini CLI 的免费和开放策略。

还有一个共同点值得强调：

> `token 或预算可见性` 必须做。

对用户来说，长期使用 Agent 最大的不安之一，就是“不知道它到底在花我多少钱”。预算展示、额度提醒和主动控制能力，不是运营细节，而是信任设计的一部分。

**本节小结**

商业模式不是后置问题，它本身也会反过来塑造用户和 Agent 的关系。用户是“顺带用”“认真买”还是“先试再说”，会直接影响他们后续的使用深度和预期。

---

### 3.4 补充亮点

> **本节结论**
> 真正影响长期使用意愿的，往往不只是主干能力，而是那些让协作更顺、更稳、更可控的小设计。

以下几个细节比较值得单独记一下：

- `Codex` 的 chronicle 录屏生成记忆：把用户做过的事情转成后续上下文，减少“之前做过但 Agent 不知道”的断层。
- `Claude Code` 的 `/recap`：跨时段继续协作时很好用，能快速把人带回之前的任务上下文。
- `Claude Code` 的 hooks：价值不在“能插逻辑”，而在于它把“调 prompt”升级成了“调工作方式”。
- `Gemini CLI` 的 `/memory show`：适合排查“它为什么这么做”，把规则透明化做得很好。
- `Gemini CLI` 的 checkpointing：很适合长任务探索，回退成本低。

**本节小结**

这些能力未必是首页最显眼的卖点，但很可能是用户真正决定留下来的原因。因为它们直接作用在“协作摩擦”上。

---

## 4、总结：好的 Agent 体验是什么

> **本章结论**
> 好的 Agent 体验，本质上不是“功能够多”或者“模型够强”，而是用户在一次次协作后，越来越愿意把事情交给它，而且交得更放心。

如果把前面的分析收拢起来，我会把“好的 Agent 体验”概括成一句话：

> 它是一段值得投入、能持续积累、并且越合作越顺的协作关系。

### 1. 一个相对清晰的体验范式

从桌面端和 CLI 的对比来看，Agent 体验可以用一个简单框架来理解：

`五个阶段 × 四类信任 × 两种能力`

先看五个阶段：

| 关系阶段 | 用户在判断什么 | 好体验的标志 |
| --- | --- | --- |
| 第一印象 | 我能不能快速理解你 | 定位清楚，上手门槛低 |
| 任务测试 | 我能不能控制你 | 过程透明，边界明确，可随时介入 |
| 结果验收 | 你交得好不好 | 结果可读、可用、可调整 |
| 问题迭代 | 你能不能按我的意思修 | 沟通成本低，修改稳定，有版本留痕 |
| 长期协作 | 值不值得继续投入 | 越用越懂我，成本可见，收益可积累 |

再看四类信任：

- `便利性`：我能不能方便地在现有工作流里用到你
- `默契度`：你会不会越来越理解我
- `可信度`：你的结果我敢不敢接
- `托付度`：我能不能把更大的任务交给你

最后是两种能力：

- `硬实力`：模型、工具、规划、执行、结果质量
- `软实力`：沟通方式、反馈节奏、边界感、情绪摩擦控制

这三层叠在一起，才比较接近一个完整的 Agent 体验。

### 2. 对 Agent 设计的三个建议

#### 建议一：先定义“协作者是谁”，再定义功能

不要一上来就堆能力，而是先想清楚：

- 这个 Agent 在用户心里应该像谁
- 它是省心型、专业型，还是开放型
- 它在不同触点上的语气、默认行为和权限边界是否一致

很多体验问题，最后都不是功能缺失，而是“人设不一致”。

#### 建议二：把过程透明和控制权当成底层设施

用户真正焦虑的，往往不是 Agent 不够聪明，而是：

- 我不知道它现在在干什么
- 我不知道它什么时候会越界
- 我不知道它到底花了我多少资源

所以权限、留痕、回退、预算可见性，应该被当成基础设施，而不是附属功能。

#### 建议三：把记忆和调教沉淀做成护城河

模型能力会快速追平，但用户在产品里沉淀下来的工作方式、偏好、规则和团队资产，不会那么快被替代。

真正有长期价值的，不是“这次回答得很好”，而是：

- 用户教过一次的东西，下一次还在
- 个人经验可以升级为团队资产
- 用户能看见、修改、继承这些沉淀

这决定了产品能不能从“好用一次”走到“很难换掉”。

### 3. 回到 DeepSeek

如果把这份调研落回到 DeepSeek 后续的 Agent 设计，我的判断是：

1. `战略上`，需要尽早明确目标用户和差异化位置，不建议直接复制头部产品的同一路线。
2. `体验上`，建议把“关系设计”作为主线，尤其重视软实力层面的细节，这会是中文场景下比较有机会拉开差距的地方。
3. `商业上`，前期可以适当降低试错门槛，中期再逐步建立更清晰的价值定价逻辑。
4. `护城河上`，应该尽早把记忆、规则、调教资产产品化，而不是只停留在 prompt 层。

最后收回到全文最核心的一句话：

> Agent 的壁垒，不只是能力更强，而是能不能让用户愿意一次次把事情交给它，并且越交越放心。

这也是我认为接下来所有 Agent 产品都会面对的同一个考题。
