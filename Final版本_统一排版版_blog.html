Agent 竞品调研

Echo·2026.5.12

💡 引言

当模型开始真正具备工具调用、长链路推理、环境感知、任务拆解与持续执行等能力之后，“Agent”终于不再只是一个停留在概念层面的词，而开始真正成为一种可以规模化落地的产品形态。

于是我们会看到，几乎所有头部模型公司都在推进自己的 Agent 产品。与此同时，大量创业公司也开始围绕 Agent 重构产品形态、交互方式与商业逻辑。

我观察到的一个很有意思的现象是，对于 Agent 产品来说，用户购买的东西，已经不再只是一个模型、一个工具、一次生成结果，而是在消费一种更长期的东西：

“持续协作关系”

用户在使用产品时，不再只是在判断“可用性”，而是会把它隐喻成一个“人”，在这个前提下去判断“我和TA能不能建立长期、良好的协作关系”

  • 你靠谱吗？
  • 你能不能替我托管任务，并且完成得让我满意？
  • 你和我之间能不能形成可积累的默契？

这也是本篇调研想要讨论的，不只是 Agent 做了哪些功能，而是一个更核心的问题：

好的 Agent 产品，到底是怎样通过“关系设计”一步步获得用户的信任，最终成为用户眼中“一个值得协作的对象”？

本文将在这个视角下重点分析三类头部产品：OpenAI Codex、Claude Code 和 Gemini CLI。我会从产品战略、体验设计和商业模式三个层面展开，也希望这份调研中的一些观察与思考，能为 DeepSeek 后续的 Agent 产品设计带来一些小小的 insight ♥️

那么下面，我们就正式进入正文吧！

1、核心竞品选择

本章先对当前 Agent 产品的主流形态进行了整体概览，并结合 DeepSeek 的公司发展路径进行对照，判断其更接近哪一类 Agent 路线，以选择最有参考价值的核心竞品。

从技术演进趋势来看，模型与其所驱动的 Agent 系统正在变得越来越不可分割。结合 DeepSeek 在近期论文更新中对模型上下文能力、复杂推理能力及多模态能力等维度所做的增强与优化，我认为 DeepSeek 未来的发展路径更可能指向对“Agent 作为模型底层原生属性”的探索，即从模型底层构建支持 Agent 运行的基础设施闭环。基于这一判断，我选取当前该赛道中的三个头部产品——OpenAI Codex、Claude Code 与 Gemini CLI——作为核心竞品进行分析。原因在于，它们均代表了“模型 + Agent 产品”的发展范式，并且分别体现了该赛道中具有代表性的三种不同技术路径，因而具有较强的对标价值与研究参考意义。

1.1 Agent产品概览

自从大模型能力从"对话生成"迈向"任务执行"（以工具调用、多步推理与长上下文能力成熟为标志）之后，AI 行业开始进入从"能力供给"走向"结果交付"的阶段：

💪 从能力上来说：以 GPT、Gemini、Claude 为代表的模型，在全局规划能力、上下文效率与原生长效记忆、环境感知与自我反思和任务粒度的精细化调度得到了深度的优化。

💰 从商业模式上来说：Agent 通过承接完整任务链路（如编程、数据分析、办公自动化等），能够显著提升用户留存与付费意愿，使 AI 产品从"工具订阅"向"生产力交付"甚至"结果付费"转变；同时，Agent 也成为模型厂商延伸价值链的关键抓手，通过用模型构建“Agent 执行复杂现实任务的能力底座”，强化生态绑定与分发能力。

而在这一趋势下，目前行业内的 Agent 形态按照其"能力栈分层"可大体被分为四类：

分类 描述 代表产品
模型 + Agent 一体化 👉 上游模型层（模型即产品） 这类产品强调「基础模型 + Agent 架构」的深度耦合设计，目标不只是提供工具，而是构建"AI 原生应用"本身，并持续向 AGI 方向进行能力与形态的演进布局。模型能力与产品体验之间几乎没有分层，二者共同构成竞争壁垒。 Gemini（Google）、OpenAI Codex（OpenAI）、Claude Code（Anthropic）
Agent 平台 👉 能力基础设施层（中台） 这类产品提供 Agent 构建与执行的基础设施，包括IDE嵌入、任务编排、工具调用、记忆与上下文管理、多 Agent 协同等核心能力，可以插入不同的模型底座。 Cursor、Windsurf、LangChain、LangGraph、Dify、Coze 等
封装式 Agent 执行系统 👉 通用执行层（产品化 Agent） 这类产品本质上是"高度封装的通用 Agent 执行系统"，通过弱化复杂配置与调参过程，强化"黑盒式执行"的体验。其核心价值在于"你只需交付任务，我负责完成"，而不是"你参与调度，我协助执行"。 Manus、Devin、Lovable 等
Agent 垂类工具 👉 场景应用层（垂直工具） 这类产品专注于在特定场景下为用户高效完成某类任务，例如信息检索、内容生成、代码补全、会议记录等。通常围绕单一或相对清晰的使用场景进行深度优化，用户以更直接的方式获得稳定、可预期的结果。 Perplexity（搜索）、Notion AI（知识管理）、Otter.ai（会议记录）

1.2 选择最合适的核心竞品

从 DeepSeek 模型的更新历史来看，无论是 DeepSeek-V3.1 对 agentic tool use 的强化，V3.2 对 “thinking” 模式与工具调用的集成，还是 V4 发布后对主流 Agent 工具的原生集成，以及其在内部 agentic coding 场景中的落地使用，都意味着 DeepSeek 正在以自研模型为核心，向完整的 Agent 产品形态延展，而不是把 Agent 能力外包给第三方平台。

基于这个判断，后面的竞品调研我会优先聚焦同属这条路线的三个代表产品：OpenAI Codex、Claude Code 和 Gemini CLI。之所以选这三家，不只是因为它们都属于“模型 + Agent 一体化”这条大路线，更重要的是，它们虽然同属一类，但各自在模型能力驱动下展现出来的 Agent 产品形态和思路是很不一样的。

1️⃣ OpenAI Codex 更像是在把 Agent 纳入一个更完整的平台生态里，重点是一体化，以及在多个工作触点之间打通协作。
2️⃣ Claude Code 更像是在把 Agent 往“工程协作者”的方向做，强调可治理、可沉淀，也更适合嵌进组织和团队流程里。
3️⃣ Gemini CLI 则更偏向另一种开放路线：开源、低门槛、开发者友好，先把使用门槛降下来，再慢慢把生态做起来。

这样进行横向对比，Agent产品定位和发展思路拉的足够开，更容易看出这个方向的不同“玩家”到底在怎么做选择，也更容易从中拿到一些有价值的insight。

2、核心竞品分析

本章的竞品分析主要围绕产品战略定位、协作体验、商业化模式三个核心目标展开：

🧭 产品战略定位：我会从长的时间轴来看，观测其产品定位选择与关键决策，理解"在什么阶段、基于什么判断做了哪些动作"，并进一步推演其未来可能的演进方向，从而洞察 Agent 产品战略层面的发展趋势。

🎯 协作体验：回到具体场景里，用真实任务去跑，重点看 Agent 在协作中到底扮演了什么角色，又是怎么一步步和用户建立更深的协作关系的，观察哪些体验细节真正影响了用户愿不愿意继续把任务交给它，为我们的产品设计提供参考。

💸 商业化模式：看产品怎么把价值闭环搭起来，包括定价、付费转化，以及收费方式和用户感知价值之间是否匹配，思考如何设计一个既能支撑长期发展、又能驱动产品持续向用户偏好演进的商业机制。

2.1 产品战略对比

本章汇总了OpenAI Codex、Claude Code 和 Gemini CLI三家产品从发布到现在的时间轴上的产品关键动作，并基于这些动作总结产品之间存在的差异和共性，目标是为了从差异去看壁垒、从共性去看趋势。小结如下：

从产品选择差异上来看，Agent的护城河有三个方向：

  • 生态整合能力｜codeX
  • 流程嵌入和治理深度｜Claude code
  • 开放、低成本和用户规模｜Gemini CLI

从产品选择共性上来看，Agent的发展趋势有三个轴线：

  • 更联通：都在增强Agent的联通生态，产品形态向「all-in-one」演化
  • 更懂我：都在增强Agent的环境感知和记忆能力
  • 更靠谱：都在增强Agent的长任务拆解能力，并将生产环境向企业升级

2.2 产品体验对比

如引言所述，我认为Agent的体验核心不在于单纯的功能设计，而在于“关系设计”。其最终目标是通过这种关系的建立，让用户逐渐对Agent产生信任，进而愿意把重要的任务交给它，甚至接受它更深度的嵌入自己的生活。因此本章将围绕“建立协作关系”构建体验分析框架，并从桌面产品端、CLI端对三款产品端用户体验旅程进行分析，同时对比其体验的共同选择，与差异，以推导理想的Agent体验都应该关注什么。小结如下：

1）Agent产品体验的核心逻辑，本质上是在模拟“面试一个长期协作对象”的过程，目标是在「第一印象、任务测试、结果验收、问题迭代、决定长期协作」的长线周期里，从智商和情商两个维度去塑造一个值得长期协作与信赖的Agent伙伴画像。
2）从体验对比来看，不同的Agent产品塑造的伙伴画像不同：CodeX是一个高情商而且聪明贴心的朋友，很会给人情绪价值；Claude Code是一个严谨专业靠谱的同事，有的时候甚至有点古板；Gemini CLI则是一个开放透明又便宜的实习生。而这些人设定位导致同样功能下不同的产品表达和设计决策，进而衍生出差异化的产品体验。

2.2.1. 分析框架

Agent的产品体验本质上是在解决“如何建立良好的协作关系”的问题，我们需要通过“关系设计”让用户逐步形成一种感受：这是一个值得合作、值得交付任务、甚至值得长期信赖的对象。而影响关系建立的因素包括两部分：

  • 第一是硬实力「智商」：即TA能力怎么样？能不能出色的完成我托管给他的任务？
  • 第二是软实力「情商」：即TA的性格（人设）、沟通方式、反馈节奏甚至情绪价值

综合「情商」和「智商」的思考维度，我围绕“关系设计”给出了以下的体验分析框架：

2.2.2. 分析实例

基于以上的体验分析框架我会分别对三款产品，最有用户心智的两个形态桌面端产品体验和CLI产品体验进行分析

2.2.2.1 桌面端产品体验

一个值得建立关系的协作伙伴应该有这样的“画像特质”：

  • 初次握手时，让用户不被“吓跑”：简单直接、容易交流，有良好的上下兼容性
  • 任务测试时，让用户保持掌控感：情商高、思考清晰，能主动向上管理、反馈及时、响应迅速
  • 结果对焦时，让用户满意且安心：智商高结果交付的好，并负责任的持续在场解决问题
  • 迭代沟通时，让用户轻松省心：理解力强，沟通效率高，会复盘，能从反馈中学习进步，举一反三
  • 长期相处下来，让用户感受到虽然有投入但是ROI很高：性价比高、具备持续成长和深度协同的潜力，越用越默契、越用越离不开

不同的产品可以构建差异化特质，塑造体验亮点：

  • CodeX像一个“贴心聪明好相处的朋友”，协作的入门成本很低
  • Claude Code像一个“非常严谨专业但有点古板的同事”，协作的天花板很高

2.2.2.2 CLI端产品体验

三家 CLI 产品的伙伴画像总结

Codex CLI

  • 像一个“已经很懂事的高阶 copilot”
  • 开箱即用，上手快、过程省心、结果可用，缺点是想往深处调教时会感觉“够用但不够深”
  • 适合定制化要求较低，要求高效快速出不错结果的场景

Claude Code

  • 像一个“可以被深度雇佣的工程同事”
  • 对开发者来说，它最突出的地方是可控制的细粒度很高，可编辑、可定制的空间也很大。你愿意花时间训练它、给它写规则、给它配子代理，它就会无限对齐你想要的样子
  • 适合面对复杂长任务或有高定制化需求的场景

Gemini CLI

  • 像一个“开放、便宜、不藏心思的实习生”
  • 试错成本最低、规则可见度最高、生态最开放，“它现在到底在想什么”是这三家里最透明的

相比于桌面端，CLI 差异化的体验主要体现在初次握手、任务测试、结果对焦、迭代沟通四个阶段，所以这一部分会主要针对这四个阶段来对比产品体验，重复的部分就不再赘述了。

🤝 阶段一：初次握手

产品目标：能快速读懂开发文档配置指令，迅速上手安装使用。
伙伴画像：能立刻干活的成品同事，不需要先培训三天。

基本能力：

三家都做到了"一行命令进入工作状态"——codex / claude / gemini，并且都在首启时给出最小可用的 onboarding（登录、模型选择、当前目录确认）。AGENTS.md / CLAUDE.md / GEMINI.md 三套指令文件的存在，也意味着它们都默认假设用户会"用一份文件来描述项目"，而不是依赖每次手动 prompt。

差异化竞争力：

🚀 Codex CLI 🛠 Claude Code 🌱 Gemini CLI
传递的感觉是："最顺手，产品化最完整" —— ChatGPT 账号登录或 API key，进场后界面非常克制，几乎只剩任务输入区，第一次用就能开始 传递的感觉是："最硬核，配置面最广" —— 启动后能看到模式、token、状态、历史搜索等更多状态信息，新手会觉得稍重，但开发者一眼就能感受到"这是一个专业工具" 传递的感觉是："最容易试，开源最透明" —— npx 就能跑，Google 登录还有免费 tier（60 req/min, 1000 req/day），仓库公开（Apache-2.0），开发者会有一种"放心试一下"的心理安全
维度 Codex CLI Claude Code Gemini CLI
安装与首启门槛 低，配套 ChatGPT 账号最顺 中，概念多，需要花时间理解模式与权限 极低，npx 直接跑，个人 Google 账号有免费额度

⏰ 阶段二：任务测试

产品目标：通过模型权限设置、过程透出、可中断的指令，让用户持续保持掌控感。
伙伴画像：边界感清晰、过程透明、不擅自越权的可靠同事。

基本能力：

2.3 商业模式对比

📌 先说结论：本章分析了三款产品不同的订阅模式，总结了三种不同的商业模式：

  • CodeX：OpenAI 靠 Agent 拉动 GPT 生态产品的订阅盘子，面向本来就在用 ChatGPT 的开发者，用更高的 token 性价比和“加量不加价”的感觉去稳住续费、带动升级，核心策略是通过生态放大订阅价值。
  • Claude Code：Anthropic 则是将 Agent 做成面向深度用户的高价值订阅产品，面向愿意为复杂任务、深度协作和高可控性买单的重度用户；虽然 token 更贵，但对这批人来说效果也确实更好。总结来说核心策略是 Agent 能力做得更深度、更定制、更可控，把盈利点放在深度使用和高价值用户客单上。
  • Gemini CLI：Google 则是先靠低门槛的 playground 吸引用户，再慢慢做转化。它先让更多人进来试、先养成使用习惯，再把其中一部分用户往 Google AI Pro、Code Assist、Vertex AI 这些更高价值的服务里转，核心策略是先铺生态入口，再做后端转化。

三家模式的判断与启示

CodeX 的“绑订阅”模式更适合已经有成熟订阅盘子的厂商，目标用户本质上是原本就在用 ChatGPT 的那批人。它的盈利点不在于单独把 Agent 卖多贵，而在于用 Agent 提升主订阅的续费率、用户时长和高阶套餐转化，让整个 ChatGPT 订阅盘子更稳、更值钱。

Claude Code 的“独立订阅”模式更适合把 Agent 当核心产品来做的公司，目标用户也更明确，就是愿意为生产力和可控性买单的开发者、团队和企业。它的盈利点是按更高价值的任务托管能力收费，用户越重度、越依赖、越深度嵌入流程，收入就越稳定。

Gemini CLI 的“开源 + 免费 tier”模式服务的是更广的开发者基数，尤其是学生、独立开发者，以及愿意先试再决定要不要付费的人。它短期看不一定靠 CLI 本身挣钱，但能把用户持续导向 Google AI Pro、Code Assist、Vertex AI 这些更高价值的产品，所以它真正的盈利点在生态后端，而不是前端这层工具本身。

2.4 还有哪些体验亮点

  • CodeX 的 chronicle 录屏自动构建 AI 记忆：解决了"我做过的事 agent 不知道"的死循环，把用户的过往工作直接转成 agent 的上下文；
  • Claude Code 的 /recap：每次回到会话先帮你"小结之前发生了什么"，跨时段连续性体验最好；
  • Claude Code 的 hooks：让用户能在 agent 执行的关键点插入自己的逻辑，是"调教 agent 工作方式"而非"调 prompt"的关键能力；
  • Gemini CLI 的 /memory show：把 agent 当前认为的规则透明可见地展示出来，是这三家里最适合 debug "agent 为什么这么做"的设计；
  • Gemini CLI 的 checkpointing：让"持续可回滚"成为日常默认，让长任务的探索成本极低。
