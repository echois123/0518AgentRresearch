# Competitor Analysis Report

这个 skill 适合处理的，不是那种“把 A/B/C 功能列一下”的浅层对比，而是要产出一份真正能支撑判断的竞品分析。

它默认服务两类目标：

- 帮用户判断应该重点看哪些竞品
- 帮用户写出一份有观点、有结构、有方法感的竞品报告

## 这个 Skill 的核心方法

默认按下面这个顺序走，不要一上来就写正文：

1. 先定义这份报告真正要回答什么问题
2. 先拉大盘，再选核心竞品
3. 不做静态功能对比，先做时间轴对比
4. 先拆“共性趋势”和“关键差异”，不要混着写
5. 分三层分析：产品战略、产品体验、商业模式
6. 体验部分优先用“关系设计/协作关系”视角来分析
7. 最后再写正文，把研究逻辑变成有判断力的报告

## 交付物

在完整模式下，这个 skill 最好产出四类内容：

- 竞品大盘与核心竞品筛选理由
- 核心竞品时间轴与关键动作表
- 共性趋势与差异壁垒的总结
- 正式竞品分析报告

如果用户只要其中一部分，也沿用同样的方法，只是缩小范围。

正式开始前，按下面顺序读取参考文件：

## Part 1：竞品分析怎么做

### 1. 先定义“这份报告到底要帮谁做什么判断”

不要直接接受“帮我分析几个竞品”这种表层任务。

先判断：

- 用户到底在做什么产品、策略判断或能力规划
- 这份报告是为了“学习”“判断路线”“找标杆”“做决策”还是“做汇报”
- 这里的竞品，应该按什么逻辑来选：直接替代品、同一工作流角色、同一技术路径，还是同一商业路径

如果这个问题没定义清楚，后面很容易写成一份信息很多但没法决策的材料。

### 2. 先拉大盘，再收窄到核心竞品

不要上来就只看用户口头点名的几个产品。

先拉一个更宽的大盘，至少包含四类对象：

- 直接竞品：解决同一个核心问题
- 邻近竞品：不完全同类，但用户会拿来比较
- 同技术路线产品：底层能力或架构路径相似
- 同商业路线产品：打法、生态、包装方式相近

然后再收窄到 2 到 4 个核心竞品。筛选标准优先看：

- 和目标问题是否足够相关
- 是否真的代表一种独立的打法
- 是否对用户预期或行业方向有塑造作用

报告里要明确写清楚：

- 为什么选这几个
- 为什么没把其他产品放进核心对比

### 3. 先做时间轴，不要直接做静态截图式对比

这个 skill 的一个关键方法，是先看“不同产品在不同时间段都在做什么”，而不是直接把今天的功能页面摆在一起。

每个核心竞品都要优先整理这些时间轴信息：

- 产品/能力首次上线时间
- 关键版本迭代
- 定价或套餐变化
- 工作流入口扩张
- 平台/插件/生态开放动作
- 模型能力或执行能力升级
- 企业级、安全、治理相关动作
- 对外叙事和目标用户是否发生变化

然后回答四个问题：

- 它在不同阶段重点在押什么
- 哪些东西变了
- 哪些东西一直没变
- 这背后反映出什么产品判断

### 4. 从时间轴里拆“共性”和“差异”

不要一边讲趋势一边讲差异，容易糊成一团。

默认拆成两部分：

- 共性：多家产品都在往哪里收敛，这通常代表行业趋势、用户预期、技术必然或平台级机会
- 差异：不同产品分别在哪些地方押注不同，这通常代表壁垒方向、目标用户选择、组织能力或商业判断

一句话概括这个方法：

- 从共性看趋势
- 从差异看壁垒

### 5. 默认拆成三层分析

除非用户明确要求别的框架，否则默认都拆成这三层：

- 产品战略
- 产品体验
- 商业模式

这样做的好处是：

- 不会把体验层面的好感误写成战略优势
- 不会把价格策略误当成功能判断
- 最终更容易回到“对我有什么启发”这个问题上

### 6. 产品体验优先用“关系设计”视角来写

这个 skill 默认继承你这次报告里最有辨识度的方法：把 Agent 或产品体验理解为“关系建立”的过程。

也就是说，体验分析不只是看“好不好用”，而是看：

- 用户如何开始信任它
- 用户如何判断它值不值得交付任务
- 用户在协作过程中有没有掌控感和安全感
- 用户会不会把它当作一个值得长期合作的对象

默认可以用这五个阶段来写体验：

1. 初次握手
2. 任务测试
3. 结果对焦
4. 迭代沟通
5. 长期合作

如果是别的产品类型，也可以改阶段名称，但原则不变：看“关系是怎么被建立起来的”。

## Part 2：竞品报告怎么写

### 1. 先写判断，再给证据

正文不要写成资料堆砌。

优先采用这个表达节奏：

1. 先抛问题
2. 先给判断
3. 再给证据
4. 最后解释这对目标产品意味着什么

这会比“先列事实再硬总结”更像一个真正做过判断的产品人。

### 2. 报告写法要像产品判断，不要像百科摘要

这类报告应该读起来像：

- 有方法
- 有筛选
- 有观点
- 有判断边界

而不是：

- 什么都写一点
- 句句都对，但没有重点
- 看起来很完整，但读完不知道结论是什么

### 3. 写作口径优先“人话判断”

优先写成这种感觉：

- “真正有意思的点在于……”
- “这背后反映的是……”
- “它更像是在……”
- “这不是单纯的功能差异，而是……”
- “如果放到更长的时间轴里看……”
- “这里透露出的判断是……”

尽量少写那种非常泛、非常空的话，比如：

- 赋能
- 全方位
- 深度融合
- 行业趋势明显

除非后面你真的解释了机制。

### 4. 体验部分不要泛泛谈 UI，而要写“协作感”

尽量避免只写：

- 布局更清晰
- 按钮更明显
- 信息架构更好

更好的写法是回答这些问题：

- 它让用户一开始觉得自己面对的是一个什么样的对象
- 它是怎么降低理解成本的
- 它在执行过程中有没有主动证明自己靠谱
- 用户要介入时，边界是否清晰
- 结果出来之后，用户是否容易确认、校准、继续推进
- 长期来看，用户是否会觉得投入它是有复利的

### 5. 结尾一定要回到“对目标产品的启发”

报告最后不能只是复述谁强谁弱。

必须回答：

- 哪些已经是行业共识
- 哪些才是真正的差异化
- 哪条路线更值得跟
- 对目标产品应该学什么、避开什么、优先补什么

## 来源规则

如果报告里涉及技术路线、模型能力、系统架构、论文方法或关键能力判断，优先使用一手资料，默认顺序如下：

1. 原始论文 / 官方技术报告 / 官方系统卡
2. 官方文档 / 官方博客 / 官方发布说明
3. 官方演讲 / 官方仓库 / 官方 issue / 官方 changelog
4. 高质量二手解读

能看原版论文时，优先看原版论文，不要只看媒体总结。

## 最后检查

交付前至少检查这几件事：

- 有没有先解释“为什么是这几个竞品”
- 时间轴是不是真的做了，不是伪时间轴
- 共性和差异有没有分开
- 战略、体验、商业模式有没有混写
- 体验部分有没有落到“关系建立”上
- 结尾有没有回到目标产品的启发
- 语言是不是太像 AI，总结是不是太空
