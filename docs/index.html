Agent 竞品调研

💡 引言

2026 年，Agent 已经从最初的“概念热”逐渐落地，形成了可消费、可使用的产品形态：它进入真实的生产环境，参与实际业务流程，理解目标、制定计划、执行任务，最终直接交付结果。

这种从「提供思路、给出建议」向「产生行动、交付结果」的职能转变，改变了用户对 Agent 的角色预期：用户不再只是将 TA 看作「聊天对话的搭子」，而是把 TA 当作「共同完成工作的协作伙伴」。相应地，用户与 Agent 之间建立的也不再是浅层的问答关系，而是包含更多维度、更长链路和更多触点的深度协作关系。

在这样的关系结构下，用户对 Agent 的评价标准也随之发生变化。最终交付结果仍然重要，但已不再是唯一的判断依据；协作过程是否顺畅、是否低摩擦、是否能够逐步建立信任与默契，开始成为影响体验的重要因素。

因此我认为，用户在 Agent 产品上消费的东西，正在从结果转变为更长期的“持续协作关系”；未来好的 Agent 产品，也不应该只是一个“结果型工具”，而应当是一个“关系型伙伴”。这也是我在本篇调研中想要重点讨论的：

如何围绕“建立长期协作关系”的过程本身进行 Agent 产品体验设计

本文将在这个视角下重点分析两个头部产品：OpenAI Codex 和 Claude Code。我会从产品战略和产品体验两个层面展开，也希望这份调研中的一些观察与思考，能为 DeepSeek 后续的 Agent 产品设计带来一些小小的 insight ♥️。

那么下面，我们就正式进入正文吧！

1 核心竞品选择

本章首先对当前 Agent 产品的主流形态进行整体梳理，并结合 DeepSeek 的公司发展路径，判断其更接近哪一类 Agent 路线，以此筛选出最具参考价值的核心竞品。从技术演进趋势来看，模型与其所驱动的 Agent 系统正在变得越来越不可分割。结合 DeepSeek 近期在上下文能力、复杂推理能力及多模态能力等方面的持续增强与优化，我认为其未来更可能指向“Agent 作为模型原生能力”的探索方向，即从模型底层构建支持 Agent 运行的基础设施闭环。基于这一判断，我选取相近赛道中的两个头部产品 OpenAI Codex 和 Claude Code 作为核心竞品进行分析。二者都代表了“模型 + Agent 产品”的发展范式，但在技术路径、产品形态与协作体验上又呈现出明显分化，因此具备较强的对标价值与研究意义。

1.1 Agent 产品概览

概览当前较为热门的 Agent 产品，我将其按照不同“能力栈分层”划分为四个类型：

分类 描述 代表产品
模型+Agent一体化
👉 上游模型层（模型即产品）		 这类产品强调「基础模型 + Agent 架构」的深度耦合设计，目标不只是提供工具，而是构建 AI 原生应用本身，并持续向 AGI 方向进行能力与形态的演进布局。模型能力与产品体验之间几乎没有分层，二者共同构成竞争壁垒。 OpenAI Codex、Claude Code、Gemini CLI
Agent平台
👉 能力基础设施层（中台）		 这类产品提供 Agent 构建与执行的基础设施，包括 IDE 嵌入、任务编排、工具调用、记忆与上下文管理、多 Agent 协同等核心能力，可以插入不同的模型底座。 Cursor、Windsurf、LangChain、LangGraph、Dify、Coze 等

1.2 选择最合适的核心竞品

从 DeepSeek 模型的更新历史来看，无论是 DeepSeek-V3.1 对 agentic tool use 的强化，DeepSeek-V3.2 对“thinking”模式与工具调用的集成，还是 V4 发布后对主流 Agent 工具的原生集成，以及其在内部 agentic coding 场景中的落地使用，都意味着 DeepSeek 正在以自研模型为核心，向完整的 Agent 产品形态延展，而不是把 Agent 能力外包给第三方平台。基于这个判断，后面的竞品调研我会优先聚焦同属这条路线的两个代表产品：OpenAI Codex 和 Claude Code。它们虽然路径方向相似，但各自在模型能力驱动下展现出来的 Agent 产品形态和思路是很不一样的。这样进行横向对比，Agent 产品定位和发展思路拉得足够开，更容易看出这个方向的不同“玩家”到底在怎么做选择，也更容易从中拿到一些有价值的 insight。

维度 CodeX Claude Code
核心定位 更像是把 Agent 纳入一个更完整的平台生态里，重点是一体化，以及在多个工作触点之间打通协作。 更像是在把 Agent 往“工程协作者”的方向做，强调可治理、可沉淀，也更适合嵌进组织和团队流程里。
价值重心 生态整合能力更强，强调广覆盖、低门槛和 all-in-one 式工作流承接。 流程嵌入与治理深度更强，强调高质量执行、长期调教和组织级信任。

2 核心竞品分析

2.1 产品战略对比

本章从时间轴与关键动作出发，汇总 OpenAI Codex 与 Claude Code 从发布到现在的产品演进（见附录1），并总结其关键决策的差异与共性，从差异去看壁垒、从共性去看趋势。小结如下：

🏬 从产品选择差异上来看，Agent 的“护城河”有两个方向：

生态整合能力（广）｜CodeX
流程嵌入和治理深度（垂）｜Claude Code

🚇 从产品选择共性上来看，Agent 的发展趋势有三个重要轴线：

更联通：都在增强 Agent 的联通生态，产品形态向「all-in-one」演化
更懂我：都在增强 Agent 的环境感知能力，并让记忆向“可持续的系统能力”演进
更靠谱：都在增强 Agent 的任务拆解能力，并将生产环境向企业标准升级

2.1.1 差异化选择

  • CodeX 走的是“平台化 AGI 入口”路线，更相信生态整合能力
  • Claude Code 走的是“企业级数字员工”路线，更相信流程嵌入和治理深度
维度 CodeX Claude Code
产品定位 以云端为核心，面向更完整的 AGI 产品形态延展。 强调治理、能力和组织协作的一体化，像“可管理的数字员工”。
目标用户 从个人开发者到企业团队的广泛用户。 高价值企业客户、重度开发者、治理要求高的组织。
初始形态 云端平台，与 ChatGPT 深度联动。 终端工具，与本地工作环境深度融合。
演进方向 扩大生态覆盖面，打通终端、IDE、Web、移动、GitHub 等触点。 提升自主性、可治理性和组织嵌入能力，成为工程体系中的正式角色。

2.1.2 共性趋势

  • 更联通：都在增强 Agent 的联通生态，产品形态向「all-in-one」演化
  • 更懂我：都在增强 Agent 的环境感知能力和工作记忆能力
  • 更靠谱：都在增强 Agent 的长任务拆解能力，并将生产环境向企业升级
共性趋势 1｜更联通
👐 关键动作：增强 Agent 的联通性生态
Agent 正在从“一个独立的对话入口”演化为“一个统一代理 + 多个工作触点”的形态，逐步嵌入 IDE、代码托管平台、项目管理工具、协作平台、自动化系统，甚至 iOS 等移动端入口中，做到更无缝地接入用户已有的工作流、贯穿更多工作触点。同时，插件、MCP、skills 等生态也在不断补齐，让 Agent 能通过更开放的接口体系持续扩展能力边界。
🧒 用户体验：和 TA 协作的迁移成本降低了，可扩展的潜力提升了
用户无需频繁切换工具、搬运上下文或迁移各类文件，就能将 Agent 无缝嵌入当前工作流中获得增量支持，0 到 1 的迁移成本显著降低，可扩展性和自定义自由度显著提升，用户进而更容易与 Agent 建立长期协作关系，而不只是将其视为一次性工具。
共性趋势 2｜更懂我
👐 关键动作 1：增强 Agent 的环境感知能力 👐 关键动作 2：记忆向“可持续系统能力”转化
相比只理解 prompt 本身，Agent 正在进一步理解用户所处的真实工作环境。它不仅要读取仓库、理解文件结构、感知当前运行环境和项目状态，还要在必要时接入终端、浏览器甚至 computer use 去观察“屏幕上正在发生什么”，从多种信息源拼出更完整的现场，再结合上下文做出更有针对性的判断，真正做到基于现场做决策、基于场景给方案。 Agent 的记忆正在从“临时对话中的上下文”演变为“可持久、可分层、可压缩、可恢复、可管理的工作记忆系统”。也就是说，记忆正在从模型内部能力，逐步转向更像产品层和系统层的能力，并被进一步封装成可编辑、可配置、可自定义的系统能力。
🧒 用户体验：和 TA 协作的沟通成本降低了，默契程度提升了
用户可以用更少的解释成本，获得更贴合当前项目场景和上下文的、更准确有针对性的解决方案；同时让用户在不同任务和场景下拥有更强的掌控感与信任感，也更有利于建立稳定的协作关系。
共性趋势 3｜更靠谱
👐 关键动作 1：更接近企业级的生产环境 👐 关键动作 2：增强复杂长任务拆解与执行能力
Agent 正在从“适合 vibe coding 的辅助工具”走向真正的企业级生产环境参与者。这意味着它不能只会生成代码，还要在代码质量、权限边界、审计追踪、审批机制、可观察性和安全治理等方面，逐步达到真实工程系统的标准，从而嵌入真实的生产流程中。 Agent 正在转向更贴近真实开发流程的工作方式：面对复杂长任务，它不再依赖单次交互完成，而是要先建立任务目标与约束，再做阶段性计划、子任务拆解、工具选择、执行排序、过程检查、结果验证与异常回退，并在中途根据新信息持续重规划。也就是说，长任务能力不只是“会拆解”，而是形成了一整套更接近工程执行的闭环。
🧒 用户体验：和 TA 协作的不确定性减少了，标准化程度提升了，更有安全感
用户不再只是获得 demo 级别的结果，而是能得到接近生产级别、可直接进入工程流程的交付物。同时当任务拆解和循环验证流程越标准化，用户越能感受到 Agent 对任务的稳定托管和持续推进，进而提升对执行过程和结果的信任。

2.2 产品体验对比

如引言所述，我认为 Agent 的体验核心不在于单纯的功能设计，而在于“关系设计”。其最终目标是通过这种关系的建立，让用户逐渐对 Agent 产生信任，进而愿意把重要的任务交给它，甚至接受它更深度地嵌入自己的生活。因此本章将围绕“建立协作关系”构建体验分析框架，并从桌面产品端、CLI 端对两款产品的用户体验旅程进行分析。小结如下：

1）Agent 产品体验的核心逻辑，本质上是在模拟“面试一个长期协作对象”的过程。好的 Agent 产品，应该能够在「第一印象、任务测试、结果验收、问题迭代、决定是否长期协作」这一长线周期中，从「智商」和「情商」两个维度共同塑造出一个值得信赖、值得长期协作的 Agent 伙伴形象。

2）从体验对比来看，不同 Agent 产品所塑造的伙伴形象也各不相同：CodeX 更像一个高情商、聪明贴心、能够提供情绪价值的朋友；Claude Code 更像一个严谨、专业、靠谱，但有时略显古板的同事。正是这些不同的人设定位，影响了相同功能在不同产品中的表达方式和设计决策，最终形成了差异化的产品体验。

2.2.1 体验分析框架

Agent 的产品体验本质上是在解决“如何建立良好的协作关系”的问题。我们需要通过“关系设计”让用户逐步形成一种感受：这是一个值得合作、值得交付任务、甚至值得长期信赖的对象。而影响关系建立的因素包括两部分：

  • 第一是硬实力「智商」：即 TA 能力怎么样，能不能出色地完成我托管给它的任务
  • 第二是软实力「情商」：即 TA 的性格（人设）、沟通方式、反馈节奏甚至情绪价值

2.2.2 体验分析实例

本章分别对比了两类产品在桌面端和 CLI 端的不同体验，小结如下：

  • 🤝 初次握手时，让用户不被“吓跑”：简单直接、容易交流，有良好的上下兼容性
  • ⏰ 任务测试时，让用户保持掌控感：情商高、思考清晰，能主动向上管理、反馈及时、响应迅速
  • 🍎 结果对焦时，让用户满意且安心：智商高，结果交付得好，并负责任地持续在场解决问题
  • 🏗️ 迭代沟通时，让用户轻松省心：理解力强，沟通效率高，会复盘，能从反馈中学习进步，举一反三
  • 📈 长期相处下来，让用户感受到虽然有投入但是 ROI 很高：性价比高、技能点满、具备持续成长和深度协同的潜力，越用越默契、越用越离不开

不同的产品可以构建差异化特质，塑造体验亮点：

  • CodeX 像一个“贴心聪明好相处的朋友”，协作的入门成本很低
  • Claude Code 像一个“非常严谨专业但有点古板的同事”，协作的天花板很高
2.2.2.1 桌面端产品体验分析

阶段一：初次握手

两款产品的共性在于都简单直接容易交流，人设定位明确，即不会吓跑用户，又能迅速建立用户心智。体现在产品层的体验是：页面结构层级较少，整体简洁且易于理解，只保留主功能区和辅助功能区两部分。信息组织上详略得当，用户认知负荷低。

对比差异来看，CodeX 的心智更 for 大众用户，借助大众最易理解的 GUI 降低了专业指令的使用门槛，但同时也保留终端、代码树等专业工具，具备较高的上下兼容性；相比之下Claude Code 心智则更偏开发者，保留IDE的操作结构，上下兼容度没有CodeX那么高，但即便是小白用户也能建立起“专业强大”的心理认知，能产生好奇和学习动力。

CodeX心智for大众用户 Claude Code心智for开发者
1）引导非常清晰，方便用户快速开始，对于小白用户来说非常友好，理解成本很低
 1）入口更像「开发工作台」，借鉴了 IDE（底部命令行）和 GitHub（热力图看板）

阶段二：任务测试

两款产品建立的伙伴画像共性是，「高情商，思考过程清晰，能够主动向上管理并及时反馈，响应也足够迅速」，核心目标都是为了保持用户对项目任务的掌控感。

  • 执行前：都会先向我确认一些不明确的细节，避免在信息不足的情况下直接执行。
  • 执行中：任务流程整体是透明的，支持任务拆分，因此既能看到大任务的整体进度，也能看到更细粒度的执行过程。同时，它也支持查看真实的脚本详情，整个过程基本都有留痕，便于追踪和理解。值得一提的是，两款产品都在近期更新了 computer use 的功能，让 Agent 能够帮用户操作“电脑屏幕”。
  • 卡点主动反馈确认：在遇到权限问题时，都会主动发起提问，不会越权执行。
  • 打断介入：支持任务排队，也支持中途打断并介入。

阶段三：结果对焦

两款产品的共性特点是，都在这个阶段树立了「高智商，结果交付得好，保持在场 stand by，有安全感负责任」的伙伴形象，让Agent高质量交付，并保证“持续在场感”。从产品交互层的体现是，能够交付质量不错的结果，完成后结果实时预览，并且支持选中元素进行批注编辑，方便用户在结果确认阶段快速介入，完成进一步校准。

2.2.3 更多体验亮点

1）双端联通的体验（手机端+电脑端）

无论是 Codex、Claude Code，还是更多正在成长中的 Agent 产品，都在尝试打通移动端与桌面端。它的价值不只是为 Agent 多提供一个入口，而是让 Agent 从一个需要被主动打开的工具，变成一个可以随时被召唤、随时接续、随时介入的协作对象。手机作为用户在日常生活中最高频使用的终端，让 Agent 不再只服务于“坐在工位上的工作时刻”，而是进一步进入更高频的生活触点，深度嵌入日常场景，为更多、更丰富的任务服务。

2）Memory 机制的升级优化

近期的更新中，CodeX 和 Claude Code 都升级了 memory 方面的功能，但各自的更新方向也折射出两家公司不同的 “DNA”：CodeX 的 Memory功能更新从用户产品视角出发，为用户解决体验问题；Claude 则从工程师技术视角切入，为Agent解决任务执行中的难题：

Chronicle

这是 Codex 最新更新里一个很值得关注的 memory 能力：它让 Agent 可以直接读取屏幕上下文，自动构建并丰富记忆。比如当屏幕弹出错误时，Agent 能直接识别错误信息，你不需要再复制、粘贴、重复描述一遍，它就能基于当前屏幕内容继续完成下一步操作。

这带来的不只是解释成本的显著下降，更重要的是，Agent 开始具备对用户行为的直接理解和持续学习能力。它最有价值的地方，在于对“行为”的记录方式发生了变化：用户不再需要主动总结自己的偏好和特点来喂给 Agent，而是 Agent 通过观察用户的真实操作，自行沉淀出对用户的理解。某种程度上，这让 Agent 更像一个真正的“伙伴”。

Auto Memory

Claude Code 新的记忆管理机制参考了“海马体编码”的生物机制，让 auto memory 在记忆碎片积累到一定程度时自动触发，对记忆进行清理和管理：提取近期会话中的重复模式，强化高价值关联，将记忆压缩至可控规模。这是一种非常优雅的工程化问题解决思路，能够为 Agent 带来一种神经启发式的、轻量且可扩展的记忆管理范式。

3）Computer Use

Computer Use 的本质，不是“让 Agent 会点鼠标”，而是把 Agent 的能力边界从“理解和生成”推进到“直接执行界面操作”。它真正解决的问题是：

  • 拓展了 Agent 的执行边界：当工具没有 API、流程无法被代码直接接管时，Agent 仍然可以通过图形界面继续推进任务，比如点击、输入、滚动、切换页面、识别按钮状态、判断页面是否进入下一步等。这使得 Agent 能够接住大量卡在 UI 层的现实任务，把推理真正落到可执行动作上。
  • 提升了用户体验的完整性：Agent 对任务的托管更彻底，用户需要介入操作的环节更少，从而降低了操作成本和中断感，也让 Agent 的整体体验更“智能”。

但不同产品走的是两条完全不同的路: Codex 走的是后台执行路线：它用独立虚拟光标在后台工作，不占用本机鼠标，用户可以一边做自己的事，一边让 Agent 持续推进任务。它优先依赖系统的无障碍树（AX tree）读取结构化界面信息，拿到按钮、状态、位置这些精确信号；读不到时再退回截图识别。这个方案的优点很明确：稳定、精准、可并行、适合长任务，更像一个系统级代理。缺点也同样明显：它强依赖结构化环境，一旦 AX tree 不可用，精度就会明显下滑。 Claude Code 走的是前台执行路线：它共用真实鼠标，通过“截图—识别—点击—再截图”的视觉闭环不断推进任务。它没有那么“系统”，但更像人，因而通用性更强、上限更高，尤其适合复杂界面和没有结构化接口的场景。代价则是慢、贵、重，而且每一步都要重新看一遍屏幕，效率和稳定性都不如后台型方案。

3 小结与展望

本篇调研的重点放在用户体验上，但在调研过程中我越来越清楚地感觉到，能力同样是无法绕开的一环。因此在小结中，我会先从本篇的核心用户体验维度去总结产品功能设计的第一条原则，接着基于我对一些技术知识的理解，补充第二条原则能力维度的思考。总结来说，Agent 产品功能的设计，需要从两个维度来考虑：

1）体验侧向外看，以用户为对象：Agent 怎么和用户沟通协作，才能建立一段紧密、持久的关系？

2）能力侧向内看，以 Agent 为对象：我们帮 Agent 解决什么问题，才能让它在任务面前更聪明、更能打？

3.1 从体验的角度来说，协作关系的设计是优化 Agent 产品体验的关键

观察 Codex 和 Claude Code 桌面端的体验，会发现一个共同点：它们的交互流程，都是围绕着想要塑造的“Agent 协作伙伴画像”展开的。这个画像决定了它该如何与用户沟通、协作，以及传递什么样的体感。

但两者的差异也恰恰体现在这里。Claude 很多设计还是沿着开发者工作流来的，比如命令行、文件、shell、手动执行、手动验证这些，它的核心思路是给开发者更多控制权。所以它更像一个技术搭档，强调透明度、可控性，以及用户对执行过程的参与感。

而 Codex 更强调结果导向和完整闭环。它会更主动地帮用户预览、验证、迭代，甚至直接基于视觉结果继续修改，让用户尽量少介入中间过程。包括最近对电脑操作、图片迭代、预览、记忆这些能力的强化，本质上也是在增强“自主完成任务”的体验。很多细节上的产品封装都能感觉到，它明显更偏用户视角去设计功能和交互，而不是把开发工作流直接暴露给用户。

这种差异背后，本质上其实是两种不同的协作关系设计：一个更强调“人控制工具”，另一个则更强调“人把任务托管给伙伴”。而用户对 Agent 的感知，也正是在一次次具体协作里逐渐形成的。两种「关系设计」各有优劣，我不加评判，但它们共同说明了一点：「协作关系」不是抽象的定位口号，而是在每一个触点、每一步交互中被实实在在“做出来”的。换句话说，「协作关系」是可以通过产品设计被有意塑造的。

这正是我想表达的，Agent 的产品设计不能只从功能出发，而要从“关系”开始，把“关系”做成产品体验的第一个设计层：

  • 在产品规划阶段，先问清楚“我们想构建一个什么样的协作者画像”“我们想让它和用户建立怎样的协作关系”
  • 基于这个人设，推导它在各个触点上应有的沟通方式与交互呈现，让人设贯穿始终，服务于同一个心智模型。
  • 把“用户对 Agent 的信任曲线”作为核心设计指标，不只是看 DAU 和留存，还要盯着用户在每个阶段的转化与流失点

拆开来看，协作旅程可以归纳为五个关键阶段：初次握手、任务测试、结果对焦、迭代沟通，以及进入长期协作后用户对投入产出比的持续评估。用户正是在这些触点上逐步建立对 Agent 的信任，更深度的交互、任务托管和日常嵌入才可能发生。从产品体验设计的角度，这可以提炼为“五阶段 × 双实力 × 三信任力”的体验范式，围绕这些阶段来系统性地设计协作关系。

在这五个触点上，用户始终在衡量 Agent 两侧的能力：

  • 硬实力（智商）：模型能力、工具调用、任务拆解、代码质量
  • 软实力（情商）：人设一致性、沟通节奏、过程透出、情绪价值

而持续协作中，用户对 Agent 的信任会沿着三个维度逐步建立起来：

  • 渗透力：够联通，工具生态开放，能在各种场景下随手唤起，不闭关锁国
  • 粘性：够靠谱，交付结果可预期、质量可验证，重要任务敢放心托付
  • 壁垒：够懂我，记得偏好、理解上下文，沟通成本越来越低，越用越离不开

3.2 从能力的角度来说，Harness决定Agent智能的上限

除了产品侧的用户体验，能力同样是Agent产品设计中绕不开的一环。我并不是技术背景出身，所以更习惯从跨学科的视角去理解Agent的技术能力。有一篇从认知科学切入的Agent综述，给了我一些启发。从认知科学的角度出发，提升智能体能力的一种有效途径，是通过任务表征的转换与外部化来转移认知负荷。购物清单并未扩大人类生物记忆的容量，而是将原本高难度的“回忆任务”转化为更轻松的“识别任务”，把认知负担从大脑迁移到了外部环境中。

更严格地说，这里涉及两个相互关联的机制。其一是外化（externalization），即将内部加工负担卸载到外部结构；其二是表征转换（representational transformation），即重新编码问题空间本身。在 Agent 设计中，这两种机制常共同出现：通过引入工具、改写提示或重塑动作空间，我们将原本需要纯粹内化推理的问题，转化为更适合模型求解的外部化表征，从而整体提升智能表现。

Agent 能力的底层核心是「Agent Loop」，其中有三个关键对象在协同工作：模型、挂载的组件、循环中持续产生的上下文。模型是核心推理引擎，负责思考和决策；挂载的组件（工具、技能等）承担外化后的认知功能，负责具体执行；而每一次交互产生的上下文，则承载着任务状态、反馈结果和记忆历史轨迹。三者的关系是：模型根据上下文做出判断，调用组件执行动作，执行结果反馈回来更新上下文，再进入下一轮循环——三者在这个闭环中持续流转，共同推进任务完成。

治理框架（Harness）的角色，就是在这条闭环之上提供一个运行时治理层，系统性地优化和规范这一循环——驱动、调整甚至编排Agent Loop，让它运转得更稳、更省、更可信，而不是把所有压力都堆在模型的推理能力上。更进一步，像控制论系统那样，让整个系统具备受控、可预测、可审计且自我进化的能力。

而围绕 Agent Loop 最通用的形态，底层逻辑始终是同一件事：把循环中仍在依赖模型“自觉”或“理解”的环节，逐步外化为确定性的工程机制。沿着这个方向，我想用设计师的视角去看循环里各个角色间的“触点”，思考Harness如何接管这些触点上的状态流动与决策边界，这也许就是harness未来的一些可做工的地方：

  • 触点一：Model ↔ Context（注意力分配与状态感知） 模型并不真正“理解世界”，它只理解当前注意力窗口中被组织后的上下文。因此，这个触点的核心问题是：在有限的注意力预算下，模型应该看到什么、忽略什么，以及如何持续感知正确的任务状态与环境状态。Harness 切入的关键点，是对上下文进行工程化治理：包括上下文分层、注意力分配、状态同步与记忆调度，帮助模型在复杂循环中始终基于正确的“世界状态”进行推理与决策。

  • 触点二：Model ↔ tool（意图到动作的映射） 工具使用的核心难点，在于如何将语言层面的开放式意图，稳定地转译为确定性的系统动作，并控制调用过程中的成本、副作用与安全风险。Harness 切入的关键点，是对“意图 → 动作”的映射过程进行约束与治理：通过工具路由、参数校验、策略编排、权限控制与沙箱机制，让工具调用从“模型自由决策”逐步演化为“策略约束下的可靠执行”。

  • 触点三：Tool ↔ Context（执行结果的情境化注入） 工具返回的原始结果通常噪声巨大、结构混乱，无法直接回灌到上下文中，否则会持续污染模型的注意力与状态理解。Harness 切入的关键点，是对执行反馈进行结构化处理：将原始输出压缩为高密度状态摘要，提取关键环境变化与任务结果，并对执行轨迹进行结构化留痕，从而支持后续推理、调试、复盘与长期优化。

  • 触点四：Context ↔ Context（记忆的长期演化与治理） 当 Agent 长期运行后，记忆系统本身会逐渐成为影响性能与行为稳定性的核心变量。记忆可能出现冗余、漂移、冲突甚至错误强化，因此问题不再只是“是否记住”，而是“如何持续演化与治理记忆”。Harness 切入的关键点，是构建 Agent 的认知治理系统：负责记忆的压缩、筛选、更新、遗忘与长期一致性维护，让记忆能够随着任务与环境变化持续自适应演化，而不是无限堆积。

  • 触点五：Model ↔ Model（多智能体协同） 当多个 Agent 共同协作时，问题不再是单个模型如何推理，而是多个模型之间如何对齐目标、共享状态与协同决策。Harness 切入的关键点，是任务编排与协同治理：负责跨 Agent 的状态共享、任务分发、角色协调、共识仲裁与结果聚合，降低多智能体系统中的推理震荡、目标冲突与协作失控问题。

  • 触点六：Tool ↔ Tool（工具链依赖与副作用隔离） 当 Agent 开始组合多个工具完成复杂任务时，工具之间会形成数据依赖、状态依赖与副作用传播，一个环节的异常可能沿链路级联扩散。Harness 切入的关键点，是将隐式的工具协作关系工程化为可治理的工作流系统：负责工具链的任务编排、状态同步、依赖管理、失败恢复、权限隔离与执行边界控制，让 Agent 从单次 Tool Use 演化为稳定、可恢复、可治理的 Runtime Workflow。
