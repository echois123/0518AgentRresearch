## **💡 引言**

2026 年，Agent 已经从最初的“概念热”逐渐落地，形成了可消费、可使用的产品形态：它进入真实的生产环境，参与实际业务流程，理解目标、制定计划、执行任务，最终直接交付结果。

这种从「提供思路、给出建议」向「产生行动、交付结果」的职能转变，改变了用户对 Agent 的角色预期：用户不再只是将 TA 看作「聊天对话的搭子」，而是把 TA 当作「共同完成工作的协作伙伴」。相应地，用户与 Agent 之间建立的也不再是浅层的问答关系，而是包含更多维度、更长链路和更多触点的深度协作关系。

在这样的关系结构下，用户对 Agent 的评价标准也随之发生变化。最终交付结果仍然重要，但已不再是唯一的判断依据；协作过程是否顺畅、是否低摩擦、是否能够逐步建立信任与默契，开始成为影响体验的重要因素。

因此我认为，用户在 Agent 产品上消费的东西，正在从结果转变为更长期的“持续协作关系”；未来好的 Agent 产品，也不应该只是一个“结果型工具”，而应当是一个“关系型伙伴”。这也是我在本篇调研中想要重点讨论的：

> 如何围绕“建立长期协作关系”的过程本身进行 Agent 产品体验设计

本文将在这个视角下重点分析两个头部产品：OpenAI Codex 和 Claude Code。我会从产品战略和产品体验两个层面展开，也希望这份调研中的一些观察与思考，能为 DeepSeek 后续的 Agent 产品设计带来一些小小的 insight ♥️。

那么下面，我们就正式进入正文吧！

## **1 核心竞品选择**

本章首先对当前 Agent 产品的主流形态进行整体梳理，并结合 DeepSeek 的公司发展路径，判断其更接近哪一类 Agent 路线，以此筛选出最具参考价值的核心竞品。从技术演进趋势来看，模型与其所驱动的 Agent 系统正在变得越来越不可分割。结合 DeepSeek 近期在上下文能力、复杂推理能力及多模态能力等方面的持续增强与优化，我认为其未来更可能指向“Agent 作为模型原生能力”的探索方向，即从模型底层构建支持 Agent 运行的基础设施闭环。基于这一判断，我选取相近赛道中的两个头部产品 OpenAI Codex 和 Claude Code 作为核心竞品进行分析。二者都代表了“模型 + Agent 产品”的发展范式，但在技术路径、产品形态与协作体验上又呈现出明显分化，因此具备较强的对标价值与研究意义。

### **1.1 Agent 产品概览**

### **2.2 产品体验对比**

如引言所述，我认为 Agent 的体验核心不在于单纯的功能设计，而在于“关系设计”。其最终目标是通过这种关系的建立，让用户逐渐对 Agent 产生信任，进而愿意把重要的任务交给它，甚至接受它更深度地嵌入自己的生活。因此本章将围绕“建立协作关系”构建体验分析框架，并从桌面产品端、CLI 端对两款产品的用户体验旅程进行分析。小结如下：

1）Agent 产品体验的核心逻辑，本质上是在模拟“面试一个长期协作对象”的过程。好的 Agent 产品，应该能够在「第一印象、任务测试、结果验收、问题迭代、决定是否长期协作」这一长线周期中，从「智商」和「情商」两个维度共同塑造出一个值得信赖、值得长期协作的 Agent 伙伴形象。

2）从体验对比来看，不同 Agent 产品所塑造的伙伴形象也各不相同：CodeX 更像一个高情商、聪明贴心、能够提供情绪价值的朋友；Claude Code 更像一个严谨、专业、靠谱，但有时略显古板的同事。正是这些不同的人设定位，影响了相同功能在不同产品中的表达方式和设计决策，最终形成了差异化的产品体验。

#### **2.2.1 体验分析框架**

Agent 的产品体验本质上是在解决“如何建立良好的协作关系”的问题。我们需要通过“关系设计”让用户逐步形成一种感受：这是一个值得合作、值得交付任务、甚至值得长期信赖的对象。而影响关系建立的因素包括两部分：

- 第一是硬实力「智商」：即 TA 能力怎么样，能不能出色地完成我托管给它的任务
- 第二是软实力「情商」：即 TA 的性格（人设）、沟通方式、反馈节奏甚至情绪价值

综合「情商」和「智商」的思考维度，我围绕“关系设计”给出了以下的体验分析框架：

| 用户的体验旅程 | 「第一印象」 | 「任务测试」 | 「结果验收」 | 问题迭代 | 长期协作 |
| --- | --- | --- | --- | --- | --- |
| 用户的心理模型 | 我能不能快速理解 TA 的价值？ | 我敢不敢把任务交给 TA 完成？ | TA 交付的结果是否符合我的期望？ | TA 能不能对齐我的改进意见？ | 和 TA 长期协作的 ROI 怎么样？ |
| 用户期望的伙伴画像 | 目标明确、专业可信、容易理解 <br>👉<br>_让我一眼就知道 TA 能帮我做什么_ | 过程可见、进度可知、边界清晰、主动反馈、工作留痕<br>👉<br>_让我始终处于可感知、可介入、可回溯的协作状态中_ | 结果可见、可用、可被接受和调整<br>👉<br>_让我处于有掌控感、可验证、可继续推进的状态_ | 修改边界清晰（知道什么该改什么不该）、修改版本留痕<br>👉<br>_让我拥有对修改过程和结果的掌控感和安全感_ | 能做到 all-in-one（嵌入各个场景工作流为我提供增量）、能懂得我理解我、token 消耗合理 <br>👉<br>_让我感觉每一次投入都在累积长期价值_ |
| 产品体验关注点 | • TA 的人设是否清晰好理解<br>• TA 看起来是不是合作性高、有礼貌也听话 | • TA 的任务理解度和听话程度如何<br>• TA 的执行进程是否透明可见<br>• TA 的执行进度是否可感知<br>• TA 能否识别卡点问题，并主动确认反馈<br>• TA 懂不懂工作留痕 | • TA 的交付物可读性如何<br>• TA 的交付物内容是否完整准确可用<br>• TA 的交付物是否会考虑到 plan b、plan c | • 和 TA 对齐的沟通成本如何<br>• TA 进行修改的准确性怎么样<br>• TA 的修改有没有归档、复盘和自我优化 | • 长期和 TA 合作我投入的成本有多高<br>• 长期和 TA 合作我获得的回报如何 |
| 产品转化流失点 | • 根本读不懂，无法快速判断是否能帮我解决问题<br>• 感觉驾驭不了，从入门到放弃 | • 我没办法控制 TA<br>• 过程黑盒，我不知道 TA 现在在做什么<br>• 进度不及时告知，我很焦虑<br>• 没有留痕追踪，我不敢信任结果 | • 交付物可读性差，无法判断效果<br>• 交付物不符合预期，怀疑 TA 的能力<br>• 考虑问题不够全面，不够聪明 | • 沟通成本太高，需要说十遍<br>• 一改就偏离原意，很失控<br>• “售后”体验差，无法回滚，同样问题还会再出现 | • token 消耗大，成本太高<br>• 联通性差，我的迁移成本很高<br>• 加分项不够多（比如个性化、结果爆好） |

#### **2.2.2 体验分析实例**

本章分别对比了两类产品在桌面端和 CLI 端的不同体验，小结如下：

- 🤝 初次握手时，让用户不被“吓跑”：简单直接、容易交流，有良好的上下兼容性
- ⏰ 任务测试时，让用户保持掌控感：情商高、思考清晰，能主动向上管理、反馈及时、响应迅速
- 🍎 结果对焦时，让用户满意且安心：智商高，结果交付得好，并负责任地持续在场解决问题
- 🏗️ 迭代沟通时，让用户轻松省心：理解力强，沟通效率高，会复盘，能从反馈中学习进步，举一反三
- 📈 长期相处下来，让用户感受到虽然有投入但是 ROI 很高：性价比高、技能点满、具备持续成长和深度协同的潜力，越用越默契、越用越离不开

不同的产品可以构建差异化特质，塑造体验亮点：

- CodeX 像一个“贴心聪明好相处的朋友”，协作的入门成本很低
- Claude Code 像一个“非常严谨专业但有点古板的同事”，协作的天花板很高

##### **2.2.2.1 桌面端产品体验分析**

#### **阶段一：初次握手**

两款产品的共性在于都简单直接容易交流，人设定位明确，即不会吓跑用户，又能迅速建立用户心智。体现在产品层的体验是：页面结构层级较少，整体简洁且易于理解，只保留主功能区和辅助功能区两部分。信息组织上详略得当，用户认知负荷低。

#### **阶段二：任务测试**

两款产品建立的伙伴画像共性是，「高情商，思考过程清晰，能够主动向上管理并及时反馈，响应也足够迅速」，核心目标都是为了保持用户对项目任务的掌控感。

- **执行前**：都会先向我确认一些不明确的细节，避免在信息不足的情况下直接执行。
- **执行中**：任务流程整体是透明的，支持任务拆分，因此既能看到大任务的整体进度，也能看到更细粒度的执行过程。同时，它也支持查看真实的脚本详情，整个过程基本都有留痕，便于追踪和理解。值得一提的是，两款产品都在近期更新了 computer use 的功能，让 Agent 能够帮用户操作“电脑屏幕”。
- **卡点主动反馈确认**：在遇到权限问题时，都会主动发起提问，不会越权执行。

##### **2.2.2.2 CLI 端产品体验分析**

#### **阶段一：初次握手**

这个阶段CLI产品应该能支持用户快速读懂开发配置文档，并且迅速上手安装使用。因此两款产品都做到了“一行命令进入工作状态”，并且都在首启时给出最小可用的 onboarding（登录、模型选择、当前目录确认）。
两款产品不同的是，一个更“傻瓜”开箱即用，另一个更“精密”配置空间大。

| 维度 | CodeX CLI | Claude Code |
| --- | --- | --- |
| 安装与首启门槛 | 低，进场快，几乎不需要理解太多概念就能开始。 | 中，概念更多，需要花时间理解模式、权限和配置面。 |
| 首启后的视觉密度 | 干净，主打“任务输入”。 | 信息密度高，主打“工作台”。 |
| 默认人设传达 | 已经很懂事的高阶 copilot。 | 严肃认真的开发代理平台。 |

#### **阶段二：任务测试**

产品在这一阶段仍旧需要保持用户对项目任务的掌控感：

- 执行前：都支持显式的权限模式选择，并在改文件、跑命令、联网等敏感动作上给出确认机制。
- 执行中：都支持流式输出过程、显示当前命令、显示生成或修改的 diff。
- 打断：都支持中断，并保留已发生的上下文，便于继续协作。

两款产品不同的点是，掌控的细粒度不同：codex更粗颗粒，三档式权限（Auto / Read-only / Full Access），把复杂的安全模型压成 3 个直觉选项，默认 Auto 时几乎不打扰，跑得很顺；Claude Code细粒度很高，权限粒度最细 —— allow / ask / deny 三态规则可叠加 sandbox、network、filesystem 维度，每个动作都能被精确治理；并支持 hooks 在执行的关键点插入用户逻辑。

| 维度 | CodeX CLI | Claude Code |
| --- | --- | --- |
| 权限模型粒度 | 更粗但更直觉，主打“省心”。 | 更细且可配置，主打“可治理”。 |
| 执行过程透出 | 简洁，结果导向。 | 更详尽，留痕和状态可见性更强。 |
| 给开发者的掌控感 | “我让你做什么你就做什么。” | “我对你做的每一件事都有发言权。” |

#### **阶段三：结果对焦**

CLI 场景下，该阶段的两款产品都需要体现留痕完整度、可观测性、是否便于纳入 CI 自动化流程，做到交付得出、查得清、改得回。两款产品都支持以 diff 形式呈现修改，也都支持非交互执行与结构化输出，便于与脚本和 CI 流程拼接。

两款产品在该阶段不同的点是对焦的精细程度不同，：

| 维度 | CodeX CLI | Claude Code |
| --- | --- | --- |
| 自动化 / CI 友好度 | 强，适合轻量 CI 流水线。 | 更强，预算、模式、代理等参数更完整，工业化能力更突出。 |
| 留痕颗粒度 | 适中，过程留痕略粗。 | 更细，适合复盘和组织协作。 |
| 交付物的“持续在场感” | 完成即等待下一指令，像安静的副驾。 | 留下完整工作记录，像有交班笔记的同事。 |

#### **阶段四：迭代沟通**

两家都支持文件化的长期指令（AGENTS.md / CLAUDE.md ）、都支持分层覆盖（项目级 / 用户级 / 全局级）、都允许在迭代中显式纠偏并写回记忆。两款产品的差异在于其调教表现和方式不同：

| 维度 | CodeX CLI | Claude Code |
| --- | --- | --- |
| 调教手段 | 规则文件简单直接，不费力就能调到一个不错的效果。 | 规则文件之外还有更多精细控制能力，几乎可以按自己的工作方式一点点把它调顺。 |
| 调教成本 | 低，上手即用。 | 高，一开始更重，但后续 ROI 更高。 |
| 调教天花板 | 中，够用但不够深。 | 高，适合复杂长任务和深度定制。 |

#### **2.2.3 更多体验亮点**

**1）双端联通的体验（手机端+电脑端）**

<img class="common-shot" src="双端联通.jpg.png" alt="双端联通" width="2616" height="1158" loading="lazy" decoding="async" />

无论是 Codex、Claude Code，还是更多正在成长中的 Agent 产品，都在尝试打通移动端与桌面端。它的价值不只是为 Agent 多提供一个入口，而是让 Agent 从一个需要被主动打开的工具，变成一个可以随时被召唤、随时接续、随时介入的协作对象。手机作为用户在日常生活中最高频使用的终端，让 Agent 不再只服务于“坐在工位上的工作时刻”，而是进一步进入更高频的生活触点，深度嵌入日常场景，为更多、更丰富的任务服务。

**2）Memory 机制的升级优化**

近期的更新中，CodeX 和 Claude Code 都升级了 memory 方面的功能，但各自的更新方向也折射出两家公司不同的 “DNA”：CodeX 的 Memory功能更新从用户产品视角出发，为用户解决体验问题；Claude 则从工程师技术视角切入，为Agent解决任务执行中的难题：

**Chronicle**

<img class="common-shot" src="Chronicle.jpg.png" alt="Chronicle" width="2616" height="1158" loading="lazy" decoding="async" />

这是 Codex 最新更新里一个很值得关注的 memory 能力：它让 Agent 可以直接读取屏幕上下文，自动构建并丰富记忆。比如当屏幕弹出错误时，Agent 能直接识别错误信息，你不需要再复制、粘贴、重复描述一遍，它就能基于当前屏幕内容继续完成下一步操作。

这带来的不只是解释成本的显著下降，更重要的是，Agent 开始具备对用户行为的直接理解和持续学习能力。它最有价值的地方，在于对“行为”的记录方式发生了变化：用户不再需要主动总结自己的偏好和特点来喂给 Agent，而是 Agent 通过观察用户的真实操作，自行沉淀出对用户的理解。某种程度上，这让 Agent 更像一个真正的“伙伴”。

**Auto Memory**

<img class="common-shot" src="Auto Memory.jpg.png" alt="Auto Memory" width="2616" height="1158" loading="lazy" decoding="async" />

Claude Code 新的记忆管理机制参考了“海马体编码”的生物机制，让 auto memory 在记忆碎片积累到一定程度时自动触发，对记忆进行清理和管理：提取近期会话中的重复模式，强化高价值关联，将记忆压缩至可控规模。这是一种非常优雅的工程化问题解决思路，能够为 Agent 带来一种神经启发式的、轻量且可扩展的记忆管理范式。

**3）Computer Use**

<img class="common-shot" src="computer use.jpg.png" alt="computer use" width="2616" height="1158" loading="lazy" decoding="async" />

Computer Use 的本质，不是“让 Agent 会点鼠标”，而是把 Agent 的能力边界从“理解和生成”推进到“直接执行界面操作”。它真正解决的问题是：

- 拓展了 Agent 的执行边界：当工具没有 API、流程无法被代码直接接管时，Agent 仍然可以通过图形界面继续推进任务，比如点击、输入、滚动、切换页面、识别按钮状态、判断页面是否进入下一步等。这使得 Agent 能够接住大量卡在 UI 层的现实任务，把推理真正落到可执行动作上。
- 提升了用户体验的完整性：Agent 对任务的托管更彻底，用户需要介入操作的环节更少，从而降低了操作成本和中断感，也让 Agent 的整体体验更“智能”。

但不同产品走的是两条完全不同的路:
Codex 走的是后台执行路线：它用独立虚拟光标在后台工作，不占用本机鼠标，用户可以一边做自己的事，一边让 Agent 持续推进任务。它优先依赖系统的无障碍树（AX tree）读取结构化界面信息，拿到按钮、状态、位置这些精确信号；读不到时再退回截图识别。这个方案的优点很明确：稳定、精准、可并行、适合长任务，更像一个系统级代理。缺点也同样明显：它强依赖结构化环境，一旦 AX tree 不可用，精度就会明显下滑。
Claude Code 走的是前台执行路线：它共用真实鼠标，通过“截图—识别—点击—再截图”的视觉闭环不断推进任务。它没有那么“系统”，但更像人，因而通用性更强、上限更高，尤其适合复杂界面和没有结构化接口的场景。代价则是慢、贵、重，而且每一步都要重新看一遍屏幕，效率和稳定性都不如后台型方案。


## **3 小结与展望**

本篇调研的重点放在用户体验上，但在调研过程中我越来越清楚地感觉到，能力同样是无法绕开的一环。因此在小结中，我会先从本篇的核心用户体验维度去总结产品功能设计的第一条原则，接着基于我对一些技术知识的理解，补充第二条原则能力维度的思考。总结来说，Agent 产品功能的设计，需要从两个维度来考虑：

1）体验侧向外看，以用户为对象：Agent 怎么和用户沟通协作，才能建立一段紧密、持久的关系？

2）能力侧向内看，以 Agent 为对象：我们帮 Agent 解决什么问题，才能让它在任务面前更聪明、更能打？

### **3.1 从体验的角度来说，协作关系的设计是优化 Agent 产品体验的关键**

观察 Codex 和 Claude Code 桌面端的体验，会发现一个共同点：它们的交互流程，都是围绕着想要塑造的“Agent 协作伙伴画像”展开的。这个画像决定了它该如何与用户沟通、协作，以及传递什么样的体感。

拆开来看，一次完整的协作用户旅程，通常会经过这几个关键触点：初次握手、任务测试、结果对焦、迭代沟通。当任务不断循环、进入长期相处后，用户会开始评估这段关系的投入产出比，它值不值得我继续投资、有没有持续的高 ROI。正是在这些触点上，用户逐步建立起对 Agent 的信任，更深度的交互、任务托管、日常嵌入才有可能发生。

所以，我认为 Agent 的产品设计不能只从功能出发，而要从“关系”开始，把“关系”做成产品体验的第一个设计层：

- 在产品规划阶段，先问清楚“我们想塑造一个什么样的协作者人设”
- 让人设贯穿所有触点，服务于同一个心智模型
- 把“用户对 Agent 的信任曲线”作为核心设计指标，不只是看 DAU 和留存，还要盯着用户在每个阶段的转化与流失点

从产品体验设计的角度，这种关系设计可以被定义为具体的「五阶段 × 双实力 × 三信任力」体验范式。

在 Agent 与用户的关系建立中，有 5 个关键触点：

| 关系阶段 | 用户在判断什么 | 对应的“好 Agent”标志 |
| --- | --- | --- |
| 第一印象 | 我能不能快速理解 TA 的价值？ | 目标明确、专业可信、容易理解 <br>👇<br> 让我一眼就知道 TA 能帮我做什么 |
| 任务测试 | 我敢不敢把任务交给 TA 完成？ | 过程可见、进度可知、边界清晰、主动反馈、工作留痕 <br>👇<br> 让我始终处于可感知、可介入、可回溯的协作状态中 |
| 结果验收 | TA 交付的结果是否符合我的期望？ | 结果可见、可用、可被接受和调整 <br>👇<br> 让我处于有掌控感、可验证、可继续推进的状态 |
| 问题迭代 | TA 能不能对齐我的改进意见？ | 修改边界清晰（知道什么该改什么不该）、修改版本留痕 <br>👇 <br>让我拥有对修改过程和结果的掌控感和安全感 |
| 长期协作 | 和 TA 长期协作的 ROI 怎么样？ | 能做到 all-in-one（嵌入各个场景工作流为我提供增量）、能懂得我理解我、token 消耗合理 <br>👇<br> 让我感觉每一次投入都在累积长期价值 |

在这五个触点上，用户始终在衡量 Agent 两侧的能力：

- **硬实力（智商）**：模型能力、工具调用、任务拆解、代码质量
- **软实力（情商）**：人设一致性、沟通节奏、过程透出、情绪价值

而持续协作中，用户对 Agent 的信任会沿着三个维度逐步建立起来：

- **渗透力**：够联通，工具生态开放，能在各种场景下随手唤起，不闭关锁国
- **粘性**：够靠谱，交付结果可预期、质量可验证，重要任务敢放心托付
- **壁垒**：够懂我，记得偏好、理解上下文，沟通成本越来越低，越用越离不开

### **3.2 从能力的角度来说，Harness决定Agent智能的上限**

除了产品侧的用户体验，能力同样是Agent产品设计中绕不开的一环。我并不是技术背景出身，所以更习惯从跨学科的视角去理解Agent的技术能力。有一篇从认知科学切入的Agent综述，给了我一些启发。从认知科学的角度出发，提升智能体能力的一种有效途径，是通过**任务表征的转换与外部化**来转移认知负荷。购物清单并未扩大人类生物记忆的容量，而是将原本高难度的“回忆任务”转化为更轻松的“识别任务”，把认知负担从大脑迁移到了外部环境中。

更严格地说，这里涉及两个相互关联的机制。其一是**外化（externalization）**，即将内部加工负担卸载到外部结构；其二是**表征转换（representational transformation）**，即重新编码问题空间本身。在 Agent 设计中，这两种机制常共同出现：通过引入工具、改写提示或重塑动作空间，我们将原本需要纯粹内化推理的问题，转化为更适合模型求解的外部化表征，从而整体提升智能表现。

Agent 能力的底层核心是「Agent Loop」，其中有三个关键对象在协同工作：模型、挂载的组件、循环中持续产生的上下文。模型是核心推理引擎，负责思考和决策；挂载的组件（工具、技能等）承担外化后的认知功能，负责具体执行；而每一次交互产生的上下文，则承载着任务状态、反馈结果和记忆历史轨迹。三者的关系是：模型根据上下文做出判断，调用组件执行动作，执行结果反馈回来更新上下文，再进入下一轮循环——三者在这个闭环中持续流转，共同推进任务完成。

治理框架（Harness）的角色，就是在这条闭环之上提供一个运行时治理层，系统性地优化和规范这一循环——驱动、调整甚至编排Agent Loop，让它运转得更稳、更省、更可信，而不是把所有压力都堆在模型的推理能力上。更进一步，像控制论系统那样，让整个系统具备受控、可预测、可审计且自我进化的能力。

而围绕 Agent Loop 最通用的形态，底层逻辑始终是同一件事：把循环中仍在依赖模型“自觉”或“理解”的环节，逐步外化为确定性的工程机制。沿着这个方向，我想用设计师的视角去看循环里各个角色间的“触点”，思考Harness如何接管这些触点上的状态流动与决策边界，这也许就是harness未来的一些可做工的地方：

- **触点一：Model ↔ Context（注意力分配与状态感知）**
模型并不真正“理解世界”，它只理解当前注意力窗口中被组织后的上下文。因此，这个触点的核心问题是：在有限的注意力预算下，模型应该看到什么、忽略什么，以及如何持续感知正确的任务状态与环境状态。Harness 切入的关键点，是对上下文进行工程化治理：包括上下文分层、注意力分配、状态同步与记忆调度，帮助模型在复杂循环中始终基于正确的“世界状态”进行推理与决策。

- **触点二：Model ↔ tool（意图到动作的映射）**
工具使用的核心难点，在于如何将语言层面的开放式意图，稳定地转译为确定性的系统动作，并控制调用过程中的成本、副作用与安全风险。Harness 切入的关键点，是对“意图 → 动作”的映射过程进行约束与治理：通过工具路由、参数校验、策略编排、权限控制与沙箱机制，让工具调用从“模型自由决策”逐步演化为“策略约束下的可靠执行”。

- **触点三：Tool ↔ Context（执行结果的情境化注入）**
工具返回的原始结果通常噪声巨大、结构混乱，无法直接回灌到上下文中，否则会持续污染模型的注意力与状态理解。Harness 切入的关键点，是对执行反馈进行结构化处理：将原始输出压缩为高密度状态摘要，提取关键环境变化与任务结果，并对执行轨迹进行结构化留痕，从而支持后续推理、调试、复盘与长期优化。

- **触点四：Context ↔ Context（记忆的长期演化与治理）**
当 Agent 长期运行后，记忆系统本身会逐渐成为影响性能与行为稳定性的核心变量。记忆可能出现冗余、漂移、冲突甚至错误强化，因此问题不再只是“是否记住”，而是“如何持续演化与治理记忆”。Harness 切入的关键点，是构建 Agent 的认知治理系统：负责记忆的压缩、筛选、更新、遗忘与长期一致性维护，让记忆能够随着任务与环境变化持续自适应演化，而不是无限堆积。

- **触点五：Model ↔ Model（多智能体协同）**
当多个 Agent 共同协作时，问题不再是单个模型如何推理，而是多个模型之间如何对齐目标、共享状态与协同决策。Harness 切入的关键点，是任务编排与协同治理：负责跨 Agent 的状态共享、任务分发、角色协调、共识仲裁与结果聚合，降低多智能体系统中的推理震荡、目标冲突与协作失控问题。

- **触点六：Tool ↔ Tool（工具链依赖与副作用隔离）**
当 Agent 开始组合多个工具完成复杂任务时，工具之间会形成数据依赖、状态依赖与副作用传播，一个环节的异常可能沿链路级联扩散。Harness 切入的关键点，是将隐式的工具协作关系工程化为可治理的工作流系统：负责工具链的任务编排、状态同步、依赖管理、失败恢复、权限隔离与执行边界控制，让 Agent 从单次 Tool Use 演化为稳定、可恢复、可治理的 Runtime Workflow。
